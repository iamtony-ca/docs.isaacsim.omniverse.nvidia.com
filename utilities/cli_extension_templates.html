CLI extension templates#

명령줄 인터페이스(CLI) 익스텐션 템플릿을 사용하면 ./repo.sh template new를 통해 터미널에서 새 Isaac Sim 익스텐션을 스캐폴딩할 수 있습니다. 네 가지 템플릿이 가장 일반적인 익스텐션 패턴을 다룹니다:

  • Python extension — extension.py, 문서, 테스트 스캐폴딩이 포함된 최소한의 Python 전용 익스텐션.

  • UI extension — Examples Browser 통합, Load World/Reset 컨트롤, 물리 콜백, 그리고 실험적 BaseSample 및 BaseSampleUITemplate 클래스를 사용한 커스텀 UI가 있는 Python 익스텐션.

  • C++ extension — Carbonite C++ 플러그인, pybind11 Python 바인딩, Python 래퍼가 있는 익스텐션.

  • OmniGraph extension — C++ 및 Python OmniGraph 노드(OGN) 정의, Carbonite 플러그인, pybind11 바인딩, OGN 빌드 통합이 있는 익스텐션.

생성기는 모든 익스텐션을 source/extensions/에 배치하며, 빌드 시스템이 자동으로 검색합니다 — 수동 등록이 필요 없습니다.

Prerequisites#

템플릿 생성기는 Isaac Sim 소스 트리에서 실행됩니다. 아직 오픈 소스 저장소를 클론하지 않은 경우 먼저 클론하세요:

git clone -b main https://github.com/isaac-sim/IsaacSim.git isaacsim
cd isaacsim

이 가이드에 표시된 모든 명령은 저장소 루트(즉, repo.sh, build.sh, templates/가 있는 디렉토리)에서 실행됩니다. Windows에서는 ./repo.sh 대신 .\repo.bat를, ./build.sh 대신 .\build.bat를 사용하세요.

./build.sh의 첫 번째 실행 시 NVIDIA Software License Agreement 동의를 요청합니다. 동의 후 저장소 루트에 .eula_accepted 마커 파일이 생성되어 다시 묻지 않습니다. 마찬가지로 ./repo.sh template new도 자체 EULA 프롬프트가 있으며 .omniverse_eula_accepted.txt를 생성합니다. 완전한 비대화형 워크플로우를 위해서는 두 파일이 모두 존재해야 합니다. 비대화형(CI) 환경에서 EULA를 사전 동의하는 방법은 아래 Non-interactive usage (CI)를 참조하세요.

플랫폼 요구사항(OS, GPU, 드라이버, 빌드 도구)의 전체 목록은 프로젝트 README를 참조하세요.

Getting started#

새 익스텐션을 생성하려면:

./repo.sh template new

대화형 프롬프트에 따라 템플릿을 선택하고 익스텐션 이름, 제목 및 기타 변수를 설정합니다.

Available templates#

Python extension#

UI, 씬 관리, C++ 코드가 필요 없는 경량 익스텐션에 사용하는 최소한의 Python 전용 익스텐션입니다.

생성된 구조:

source/extensions/isaacsim.my.extension/
├── config/extension.toml
├── data/icon.png, preview.png
├── docs/Overview.md, CHANGELOG.md
├── isaacsim/my/extension/
│   ├── __init__.py
│   ├── extension.py
│   └── tests/__init__.py, test_extension.py
└── premake5.lua

수정할 주요 파일:

  • extension.py — on_startup() 및 on_shutdown()에 시작/종료 로직을 추가하세요.

UI extension#

완전한 UI 및 씬 관리 기능이 있는 Python 익스텐션입니다. 이 템플릿은 다음을 제공합니다:

  • Examples Browser 통합 — Window > Examples Browser의 선택한 카테고리 아래에 익스텐션이 표시됩니다.

  • Load/Reset 월드 컨트롤 — 시뮬레이션 라이프사이클을 관리하는 미리 빌드된 Load World 및 Reset 버튼.

  • 물리 콜백 — SimulationManager를 통해 모든 시뮬레이션 스텝마다 호출되는 on_physics_step()이 있는 scenario.py.

  • 커스텀 UI — btn_builder 및 기타 UIElementWrapper 유틸리티를 사용하여 자신만의 버튼과 컨트롤을 추가할 수 있는 Actions 프레임이 있는 ui.py.

이 템플릿은 실험적 API를 사용합니다:

  • isaacsim.examples.base.base_sample_experimental.BaseSample — 씬 라이프사이클 관리

  • isaacsim.examples.base.base_sample_extension_experimental.BaseSampleUITemplate — UI 관리

  • isaacsim.core.simulation_manager.SimulationManager — 물리 콜백

  • isaacsim.core.experimental.utils — 스테이지 및 앱 유틸리티

생성된 구조:

source/extensions/isaacsim.my.example/
├── config/extension.toml
├── data/icon.png, preview.png
├── docs/Overview.md, CHANGELOG.md
├── isaacsim/my/example/
│   ├── __init__.py
│   ├── extension.py   ← Examples Browser registration
│   ├── scenario.py    ← BaseSample with physics callbacks
│   ├── ui.py          ← BaseSampleUITemplate with custom controls
│   └── tests/__init__.py, test_extension.py
└── premake5.lua

수정할 주요 파일:

  • scenario.py — setup_scene()에 시뮬레이션 에셋을, on_physics_step()에 로직을 추가하세요.

  • ui.py — build_extra_frames()에 커스텀 버튼과 컨트롤을 추가하세요.

C++ extension#

Carbonite C++ 플러그인과 pybind11 Python 바인딩이 있는 익스텐션입니다. 플러그인은 Carbonite 인터페이스(IExample)를 구현하고 바인딩은 acquire_example_interface() / release_example_interface()를 Python에 노출합니다. Python extension.py는 시작 시 인터페이스를 획득하여 Carbonite 플러그인이 로드되도록 트리거합니다.

생성된 구조:

source/extensions/isaacsim.my.extension/
├── config/extension.toml
├── data/icon.png, preview.png
├── docs/api.rst, Overview.md, CHANGELOG.md
├── include/isaacsim/my/extension/IExample.h       ← Carbonite interface
├── plugins/isaacsim.my.extension/ExamplePlugin.cpp ← Plugin implementation
├── bindings/isaacsim.my.extension/Bindings.cpp     ← pybind11 bindings
├── python/
│   ├── __init__.py
│   ├── impl/__init__.py, extension.py
│   └── tests/__init__.py, test_extension.py
└── premake5.lua

binding_module 변수는 pybind11 모듈 이름을 제어합니다(예: my_extension은 _my_extension.so를 생성).

수정할 주요 파일:

  • IExample.h — Carbonite 인터페이스 메서드를 정의합니다.

  • ExamplePlugin.cpp — 인터페이스를 구현합니다.

  • Bindings.cpp — pybind11을 통해 추가 메서드를 Python에 노출합니다.

  • extension.py — 인터페이스는 이미 시작 시 획득됩니다; Python 측 로직을 추가하세요.

OmniGraph extension#

C++ 및 Python OmniGraph 노드 정의, 노드 등록을 위한 Carbonite 플러그인, pybind11 바인딩, 완전한 OGN 빌드 통합이 있는 익스텐션입니다.

C++ 노드는 nodes/의 .ogn + .cpp 쌍으로 정의됩니다. Python 노드는 python/nodes/의 .ogn + .py 쌍으로 정의됩니다. OGN 빌드 시스템이 두 언어 모두를 위한 데이터베이스 클래스와 테스트 스캐폴딩을 자동 생성합니다.

Carbonite 플러그인은 carbOnPluginStartup() / carbOnPluginShutdown()에서 INITIALIZE_OGN_NODES() / RELEASE_OGN_NODES()를 구현합니다. pybind11 바인딩은 acquire_example_nodes_interface()를 노출하며, extension.py가 시작 시 이를 호출해 플러그인을 로드하고 C++ OGN 노드를 등록합니다.

생성된 구조:

source/extensions/isaacsim.my.nodes/
├── config/extension.toml
├── data/icon.png, preview.png
├── docs/api.rst, Overview.md, CHANGELOG.md
├── include/isaacsim/my/nodes/IExampleNodes.h          ← Carbonite interface
├── nodes/
│   ├── OgnExampleCpp.ogn, OgnExampleCpp.cpp           ← C++ OGN node
│   ├── config/CategoryDefinition.json
│   └── icons/isaac-sim.svg
├── plugins/isaacsim.my.nodes/PluginInterface.cpp       ← Plugin with OGN macros
├── bindings/isaacsim.my.nodes/Bindings.cpp             ← pybind11 bindings
├── python/
│   ├── __init__.py
│   ├── impl/__init__.py, extension.py
│   ├── nodes/
│   │   ├── OgnExamplePython.ogn, OgnExamplePython.py   ← Python OGN node
│   │   ├── config/CategoryDefinition.json
│   │   └── icons/isaac-sim.svg
│   └── tests/__init__.py, test_extension.py
└── premake5.lua

C++ 템플릿과 달리 OGN 템플릿은 별도의 binding_module 변수를 사용하지 않습니다. OGN 빌드 시스템이 익스텐션 이름에서 바인딩 모듈 이름을 자동으로 파생합니다(예: isaacsim.my.nodes → _isaacsim_my_nodes).

수정할 주요 파일:

  • .ogn 파일 — 노드 입력, 출력, 메타데이터를 정의합니다(JSON 형식).

  • OgnExampleCpp.cpp — C++ 노드의 compute() 메서드를 구현합니다.

  • OgnExamplePython.py — Python 노드의 compute() 메서드를 구현합니다.

  • 노드를 더 추가하려면 해당 nodes/ 디렉토리에 새 .ogn + .cpp/.py 쌍을 생성하세요.

Tutorial: Creating and testing an extension#

이 튜토리얼은 Python 익스텐션을 생성하고 빌드하여 동작을 확인하는 과정을 안내합니다.

Step 1: Generate the extension#

템플릿 생성기를 실행합니다:

./repo.sh template new

Python extension 템플릿을 선택하고 프롬프트에 값을 입력합니다:

  • Extension name: isaacsim.my.hello

  • Title: Hello World

  • Version: 0.1.0

  • Description: A hello world extension.

  • Category: Examples

다음과 같이 표시됩니다:

Extension 'isaacsim.my.hello' created successfully in
source/extensions/isaacsim.my.hello

Step 2: Build#

새 익스텐션이 포함되도록 프로젝트를 다시 빌드합니다:

./build.sh

빌드 시스템이 익스텐션을 자동으로 검색합니다 — 수동 등록이 필요 없습니다. 빌드가 성공하면 다음과 유사한 요약이 표시됩니다:

BUILD (release) SUCCEEDED (Took NN.NN seconds)

빌드 후 익스텐션은 다음 위치에 스테이징됩니다:

_build/linux-x86_64/release/exts/isaacsim.my.hello/

빌드에 실패하면 상세 출력을 보기 위해 ./build.sh -v로 다시 실행하고 첫 번째 오류 메시지를 확인하세요 — 이후 오류는 대개 첫 번째 오류의 연쇄 결과입니다.

Step 3: Run the startup test#

생성된 모든 익스텐션에는 시작 테스트가 포함됩니다. 빌드 디렉토리에서 실행합니다:

cd _build/linux-x86_64/release
./tests/tests-isaacsim.my.hello.sh

두 가지 테스트 스위트를 실행합니다:

  • 익스텐션이 활성화된 상태로 Kit를 로드하여 정상적으로 시작/종료됨을 확인하는 시작 테스트.

  • isaacsim/my/hello/tests/test_extension.py의 단위 테스트.

성공적인 실행은 표준 unittest 요약 줄로 끝납니다:

Ran N tests in N.NNs

OK

테스트가 실패하면 실패한 테스트 이름과 트레이스백이 요약 위에 출력됩니다. 실행 중 작성된 Kit 로그 파일은 ~/.nvidia-omniverse/logs/ 아래에서 찾을 수 있습니다.

Step 4: Verify in Isaac Sim#

익스텐션을 활성화하여 Isaac Sim을 시작합니다:

./_build/linux-x86_64/release/isaac-sim.sh --enable isaacsim.my.hello

Extension Manager 확인

Extensions Manager(Window > Extensions)를 열고 isaacsim.my.hello를 검색합니다. 익스텐션이 목록에 표시되고 상태 토글이 활성화(녹색)로 설정되어 있으며 extension.toml의 제목과 설명이 표시되어야 합니다.

Script Editor 확인

Window > Script Editor에서 다음을 실행합니다:

import omni.kit.app
ext_mgr = omni.kit.app.get_app().get_extension_manager()
print(ext_mgr.is_extension_enabled("isaacsim.my.hello"))
# Expected: True

Step 5: Customize#

source/extensions/isaacsim.my.hello/isaacsim/my/hello/extension.py를 열고 on_startup 및 on_shutdown 메서드를 수정하여 커스텀 로직을 추가하세요. 빌드 후 다시 테스트합니다.

Verifying UI, C++, and OmniGraph extensions#

확인 흐름은 모든 템플릿에 동일합니다(빌드 → 시작 테스트 실행 → Extensions Manager에서 시작 후 확인); 아래 추가 확인은 템플릿별 기능을 확인합니다.

UI extension

익스텐션을 활성화한 후 Window > Examples Browser를 엽니다. 생성 시 선택한 카테고리(예: Examples) 아래에 익스텐션이 표시됩니다. 선택 시 Load World와 Reset 버튼이 있는 패널이 열립니다:

  • Load World를 클릭하면 — scenario.py::setup_scene()에 정의된 기본 씬(Isaac/Environments/Grid/default_environment.usd에서 참조된 격자 지면)이 로드되고 물리 콜백이 활성화됩니다.

  • Reset을 클릭하면 — 월드가 초기 상태로 돌아가고 물리 콜백이 중지됩니다.

setup_scene() 및 on_physics_step()의 본문을 교체하여 자신만의 시뮬레이션을 구동하세요.

C++ extension

빌드 후 바인딩이 작동하는지 확인합니다:

cd _build/linux-x86_64/release
./tests/tests-isaacsim.my.extension.sh

단위 테스트는 바인딩 모듈을 임포트하고 Carbonite 인터페이스의 greet() 메서드를 호출합니다. Script Editor에서 대화형으로 확인할 수도 있습니다(자신의 extension_name과 binding_module으로 대체):

from isaacsim.my.extension.bindings._my_extension import acquire_example_interface
iface = acquire_example_interface()
print(iface.greet())
# Expected: a non-empty greeting string

OmniGraph extension

C++ 및 Python 노드 모두 등록되었는지 확인합니다:

cd _build/linux-x86_64/release
./tests/tests-isaacsim.my.nodes.sh

단위 테스트는 두 노드 타입이 og.get_registered_nodes()에 표시되고 compute() 메서드가 올바른 결과를 반환하는지 확인합니다.

Script Editor를 통해 OGN 노드를 대화형으로 확인할 수도 있습니다:

import omni.graph.core as og
nodes = og.get_registered_nodes()
print([n for n in nodes if "my.nodes" in n])
# Expected: ['isaacsim.my.nodes.ExampleCpp', 'isaacsim.my.nodes.ExamplePython']

노드를 그래픽으로 테스트하려면 Window > Graph Editors > Action Graph를 열고 새 그래프를 만든 다음 노드 팔레트에서 ExampleCpp / ExamplePython을 검색하세요. 두 노드 모두 생성된 CategoryDefinition.json 파일에 정의된 익스텐션 이름의 첫 번째 세그먼트(예: isaacsim)로 명명된 카테고리 아래에 표시됩니다.

Non-interactive usage (CI)#

CI(Continuous Integration) 자동화를 위한 전체 파이프라인은 다음과 같습니다:

  1. EULA 사전 동의. ./build.sh와 ./repo.sh template 모두 처음 사용 시 프롬프트가 표시됩니다; 비대화형 작업에서는 프롬프트가 실행을 차단합니다. CI 설정 단계에서 두 마커 파일을 생성하세요:

    touch .eula_accepted
touch .omniverse_eula_accepted.txt

    • .eula_accepted는 tools/eula_check.sh에서 확인합니다(./build.sh에서 사용).

    • .omniverse_eula_accepted.txt는 템플릿 생성기에서 확인합니다(./repo.sh template new 및 ./repo.sh template replay에서 사용).

    두 파일 모두 빌드 간에 유지되는 빈 파일이므로 작업 시작 시 한 번 생성하면 충분합니다.

    Note

    .eula_accepted만 있는 경우 ./build.sh는 작동하지만 플레이백 프론트엔드가 대화형 프롬프트에 응답할 수 없어 EULA 프롬프트에서 처리되지 않은 예외가 발생하므로 ./repo.sh template replay는 실패합니다.

  2. 플레이백 파일을 한 번 대화형으로 생성합니다. 이 단계는 템플릿 선택과 변수 값을 TOML 파일로 기록합니다:

    ./repo.sh template new --generate-playback my_extension.toml

  3. CI에서 비대화형으로 재생합니다. 재생 명령은 플레이백 파일을 사용하여 프롬프트 없이 익스텐션을 생성합니다:

    ./repo.sh template replay my_extension.toml

  4. 위 튜토리얼과 같이 빌드하고 시작 테스트를 실행합니다:

    ./build.sh
cd _build/linux-x86_64/release
./tests/tests-<extension_name>.sh

플레이백 파일은 TOML(Tom's Obvious, Minimal Language) 형식을 사용하여 템플릿과 변수 값을 지정합니다. 섹션 헤더는 템플릿 이름(isaacsim-python-extension, isaacsim-ui-extension, isaacsim-cpp-extension, isaacsim-omnigraph-extension 중 하나)를 지정합니다:

[isaacsim-python-extension]
extension_name = "isaacsim.sensors.lidar"
title = "Lidar Sensor"
version = "0.1.0"
description = "Provides lidar sensor simulation."
category = "Sensors"

templates/tests/ 디렉토리에는 CI 빌드 검증에 사용되는 미리 정의된 플레이백 파일(test_python_extension.toml, test_ui_extension.toml, test_cpp_extension.toml, test_omnigraph_extension.toml)이 있으며 자신의 파이프라인을 구성할 때 시작점으로 복사할 수 있습니다.

Build and verification checklist#

익스텐션을 생성한 후 항상:

  1. 빌드 — ./build.sh

  2. 테스트 — cd _build/linux-x86_64/release && ./tests/tests-<extension_name>.sh

C++ 및 OmniGraph 템플릿의 경우 빌드가 Carbonite 플러그인과 pybind11 바인딩을 컴파일합니다. 빌드가 실패하면 premake5.lua의 include 경로가 익스텐션의 디렉토리 레이아웃과 일치하는지 확인하세요.

Troubleshooting#

런타임에 익스텐션이 로드되지 않는 경우

config/extension.toml의 익스텐션 이름이 source/extensions/ 아래 디렉토리 이름과 일치하는지 확인하세요. 명령줄에서 --enable <extension_name>으로 또는 Extensions Manager를 통해 익스텐션이 활성화되어 있는지 확인하세요.

C++ 플러그인이 로드되지 않는 경우

extension.toml의 [[native.plugin]] 섹션에 path = "bin/*.plugin"이 포함되어 있는지 확인하세요. 없으면 Kit가 익스텐션의 bin/ 디렉토리에서 .so 플러그인 파일을 찾을 수 없습니다.

C++ OGN 노드가 등록되지 않는 경우

INITIALIZE_OGN_NODES()를 실행하려면 Carbonite 플러그인이 시작되어야 합니다. 이는 pybind11 바인딩에서 extension.py의 acquire_example_nodes_interface()가 호출될 때 발생합니다. C++ 노드가 og.get_registered_nodes()에 표시되지 않으면 다음을 확인하세요:

  • Bindings.cpp에 defineInterfaceClass<IExampleNodes> 호출이 있는지

  • extension.py가 on_startup()에서 acquire_example_nodes_interface()를 임포트하고 호출하는지

  • PluginInterface.cpp에 CARB_PLUGIN_IMPL_DEPS가 없는지(이것이 플러그인 로드를 막을 수 있음)

OmniGraph 노드 빌드 오류

.ogn 파일이 유효한 JSON인지, 각 C++ .ogn에 REGISTER_OGN_NODE() 매크로가 있는 일치하는 .cpp 파일이 있는지 확인하세요. Python .ogn 파일에는 "language": "Python"과 compute() 메서드가 있는 일치하는 .py 파일이 필요합니다.

빌드 중 Stubgen 실패

stubgen 단계가 generic_type: type "X" is already registered 오류와 함께 실패할 수 있습니다. 이는 여러 익스텐션이 타입 이름을 공유할 때 pybind11 스텁 생성기에서 발생하는 알려진 문제입니다. 빌드 자체는 성공하며 — 오직 .pyi 스텁 생성만 실패하므로 런타임에는 영향을 미치지 않습니다.

템플릿 재생 시 "This should not be encountered" 오류

저장소 루트에 .omniverse_eula_accepted.txt가 없다는 의미입니다. 템플릿 생성기의 플레이백 프론트엔드는 EULA 프롬프트에 대화형으로 응답할 수 없으므로 select() 메서드에서 예외가 발생합니다. 파일을 생성하여 해결하세요:

touch .omniverse_eula_accepted.txt

이것은 ./build.sh에서 사용하는 .eula_accepted와 별도의 파일입니다. 완전한 비대화형 CI 파이프라인에는 두 파일 모두 존재해야 합니다.
