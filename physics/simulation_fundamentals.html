Physics Simulation Fundamentals#

Physics in USD Schemas#

에셋의 물리 속성은 USD Physics Schemas와 Physx Schemas를 사용하여 잘 정의되어 있습니다. 물리 속성의 문서와 코드에서 접근하는 방법은 C++로 정의되어 있지만, 이 가이드라인을 따라 Python에서 동등한 호출을 찾을 수 있습니다.

import omni.usd
from pxr import PhysxSchema, Usd, UsdGeom, UsdPhysics

stage = omni.usd.get_context().get_stage()
prim = stage.GetPrimAtPath("/Path/To/Prim")
physics_api_prim = UsdPhysics.SomePhysicsAPI(prim)
physx_api_prim = PhysxSchema.AnotherPhysxAPI(prim)

# Check if the API is Applied, if not, Apply it.
if not physics_api_prim:
    physics_api_prim = UsdPhysics.SomePhysicsAPI.Apply(prim)

physics_attr = physics_api_prim.GetSomePhysicsAttr()
physx_attr = physx_api_prim.GetPhysxAttr()

# Check if Attribute is authored, otherwise create it
if not physics_attr:
    physics_attr = physics_api_prim.CreateSomePhysicsAttr(1.0)
print(physics_attr.Get())
physics_attr.Set(10.0)

일부 경우에는 Prim을 특정 API로 캐스팅할 때 추가 파라미터가 필요합니다. 예를 들어 Joint State는 조인트 유형("Prismatic" 또는 "Angular" 등)을 요구합니다. 이 경우 C++ 시그니처에 "TfToken" 유형이 포함됩니다. 기본 문자열로 대체하면 Python에서 작동합니다.

UI에서 보이는 물리 속성의 속성 이름을 알아야 하는 경우, 속성 패널에서 속성 위에 마우스를 올리면 툴팁에 이름이 표시됩니다. 속성 이름 표준은 schema_name:attribute_name입니다. 예를 들어 리지드 바디의 physics:velocity는 Physics Rigid Body API를 사용하고 속성 이름이 velocity임을 의미하므로, 해당 속성 게터는 UsdPhysics.RigidBodyAPI(prim).GetVelocityAttr()입니다.

Simulation Timeline#

시뮬레이션 시간은 실제 시간과 다릅니다. 시스템 구성과 시뮬레이션 환경의 크기에 따라 각 시간 스텝이 시뮬레이션하는 시간보다 빠르거나 느리게 계산되어, 결과를 순차적으로 표시하면 왜곡된 속도가 발생할 수 있습니다(Isaac Sim의 물리 시뮬레이션은 종종 실시간보다 빠릅니다). 이를 완화하기 위해 Isaac Sim은 기본적으로 실시간 속도와 일치하는 제한기가 구성되어 있습니다.

또한 시뮬레이션은 렌더링보다 빠른 속도로 실행될 수 있어, 렌더링되는 각 프레임에 대해 백그라운드에서 두 개 이상의 시뮬레이션 타임 스텝이 발생할 수 있습니다. 아래 간단한 예에서 시뮬레이션은 초당 120 타임 스텝으로 실행되고 렌더링은 초당 60 프레임으로 설정되어 렌더링된 프레임당 두 번의 물리 스텝이 발생합니다:

Note

물리 스텝 시간이 반드시 시스템 시간(시뮬레이션 시작부터)과 일치하는 것은 아닙니다. 시뮬레이션이 실시간보다 빠르게 실행될 수 있는 경우, 렌더링이나 프레임 레이트 차단 없이 타임라인에서 가속된 버전의 시뮬레이션을 실행할 수 있습니다.

이상적으로 시뮬레이션과 렌더링은 서로 일치하거나 배수 관계여야 하지만, 그렇지 않은 경우 각 렌더링 프레임에 불균등한 수의 시뮬레이션 타임스텝이 포함될 수 있습니다. 예를 들어 초당 100 스텝의 시뮬레이션, 초당 30 프레임의 렌더링에서 대부분의 렌더링 업데이트에는 3번의 시뮬레이션 스텝이 있지만 가끔 프레임에 4번이 포함됩니다.

타임라인에는 세 가지 이벤트 스트림이 있습니다(다른 것도 있지만 Isaac Sim에서 특히 관련성 높은 것들입니다). 시뮬레이션 이벤트 또는 프레임 업데이트 이벤트(렌더링 전 또는 후)에 직접 구독할 수 있습니다. OmniGraph 노드는 일반적으로 사전 렌더링 이벤트에서 업데이트되지만 매 물리 스텝 등 다른 이벤트에서 업데이트되도록 설정하는 방법도 있습니다.

Configuring Frame Rate#

스테이지의 Timecodes per second는 현재 스테이지 메타데이터를 조정하여 구성할 수 있습니다. Layer 탭에서 Root Layer를 선택하고 속성 패널에서 Timecodes per second 속성을 수정하세요. GUI 기본값 /app/player/useFixedTimeStepping=true에서 타임라인은 1 / TimeCodesPerSecond를 틱당 dt로 사용하므로, 이 값은 시뮬레이션의 벽시계 재생 속도와 기록된 애니메이션 속도를 모두 설정합니다. 물리 스텝 속도는 Physics Scene에서 별도로 구성됩니다. 세 가지 속도 간의 관계는 Architecture: Timeline, Physics, and the Renderer를 참조하세요.

Configuring Simulation Timesteps#

시뮬레이션 초당 스텝 수는 Physics Scene에서 결정됩니다. 스테이지에 Physics Scene이 없으면 기본값인 초당 60 스텝을 사용합니다.

Simulation Scene 요소를 추가하려면:

  1. Create > Physics > Simulation Scene을 클릭합니다.

  2. Simulation scene을 선택합니다.

  3. 속성 패널에서 Simulation Steps per Second 요소를 확인합니다.

Physics Scene의 다른 파라미터에 대한 자세한 내용은 Omniverse Physics Developer Guide를 참조하세요.

Simulation Components#

USD 스테이지의 Prim은 기본적으로 물리가 활성화되어 있지 않지만 UI 또는 Python 스크립트를 통해 시뮬레이션 속성을 추가할 수 있습니다. 다음은 기본 물리 오브젝트 유형을 진행하면서 요소를 추가하는 예제 씬을 생성합니다.

시작하려면 새 씬을 만들고 지면 평면을 추가하세요: File > New, 그런 다음 Create > Physics > Ground Plane.

Rigid Body#

이것은 가장 기본적인 요소입니다. 리지드 바디 역학을 추가하면 요소가 중력 가속도와 다른 외력의 영향을 받을 수 있게 됩니다.

1. 리지드 바디로 사용할 컨테이너 추가: Create > Xform. 2. 속성 패널에서 Z=10으로 이동합니다. 3. 리지드 바디로 만들려면 스테이지에서 우클릭한 다음 Add > Physics > Rigid Body.

Xform이 이제 리지드 바디임을 확인합니다. 시각적 메시가 없어 보이지 않을 수 있습니다.

Cube를 안에 중첩하여 해결할 수 있습니다:

  1. Create > Mesh > Cube를 선택하고 Xform에 드래그합니다.

  2. 큐브의 Translate가 [0,0,0.5]로 설정되어 있는지 확인합니다.

위 스크린샷과 동일한 설정을 완료한 후 Play를 눌러 결과를 확인합니다:

다음을 검토합니다:

  • 오브젝트가 떨어지면서 Z 위치가 업데이트되는 것을 확인합니다. 리지드 바디를 직접 강조 표시하고 있기 때문입니다. 큐브를 다시 선택하면 변경되지 않는 것을 확인할 수 있습니다.

  • 큐브가 지면을 바로 통과합니다. 다른 오브젝트와 충돌해야 한다는 것을 시뮬레이션에 알려야 합니다.

Colliders#

리지드 바디가 충돌하도록 하려면 시뮬레이션에 이를 원한다는 것을 표시해야 합니다. 이를 위해 Collider API가 있습니다.

  1. Cube prim을 선택하고 Add Button > Physics > Collider를 클릭합니다.

  2. 시뮬레이션을 다시 실행하여 리지드 바디가 지면에서 멈추는지 확인합니다.

콜라이더는 움직이지 않는 오브젝트에도 추가할 수 있습니다. 실험해 봅시다:

  1. 새 큐브를 만들어 Z=3.0에 배치합니다.

  2. 스케일을 [2,2,0.01]로 변경하여 2x2미터 플랫폼을 만듭니다.

  3. 리지드 바디 추가 없이 이전처럼 콜라이더를 추가합니다.

시뮬레이션을 다시 실행하여 결과를 확인합니다:

Xform 위치를 Z=80으로 올립니다. 시뮬레이션을 다시 실행합니다.

이 예제로 물리 시뮬레이션의 일반적인 문제를 해결하고 있습니다. 시간이 이산화되어 있기 때문에 오브젝트가 너무 빠르게 움직이면 한 타임스텝에서는 플랫폼 위에 있고 다음에는 완전히 통과하여 충돌이 감지되지 않습니다. 지면 평면에서는 이런 문제가 발생하지 않는데, 관통된 오브젝트를 지면 표면 방향으로 밀어내는 "포스 필드"를 구현하기 때문입니다.

이를 해결하기 위해 물리 씬에서 Enable CCD(Continuous Collision Detection)라는 옵션을 활성화하세요. CCD는 한 포즈에서 다음 포즈로 오브젝트를 스윕합니다. 이 옵션은 리지드 바디 자체에서도 활성화해야 합니다:

  1. Xform을 선택합니다.

  2. 속성 패널에서 리지드 바디 속성 아래 CCD를 활성화합니다.

이 문제를 해결하는 다른 방법도 있지만 이 시나리오에서는 이것이 가장 효과적입니다.

충돌은 화면에서 보이는 것과 관계없다는 것을 기억하세요. 예를 들어, 큐브를 숨겨도 콜라이더는 동일하게 동작하고, Xform 아래에 다른 큐브나 구를 추가해도 Collision API를 적용하지 않으면 효과가 없습니다.

많은 오브젝트 콜라이더는 여러 메시 요소의 구성으로 만들어져 형태와 동작을 결정합니다. 스테이지에서 물리적으로 분리되어 있어도 모두 리지드 바디의 자식이면 단일 리지드 바디로 작동합니다.

이 리지드 바디에 콜라이더를 추가하거나 제거하거나 씬에 더 많은 리지드 바디를 추가하여 동작을 확인해 보세요.

Convex Hull#

다음 콜라이더 실험에서는 이전에 추가한 플랫폼을 제거하고 Xform을 Z=10으로 되돌립니다.

  1. 플랫폼 자리에 Z=3.0에 Torus 메시를 추가하고 [5.0, 5.0, 5.0]으로 스케일합니다.

  2. Add > Physics > Rigid Body With Colliders Preset.

  3. 시뮬레이션을 실행합니다.

큐브가 토러스 구멍 위에 앉는 것은 메시 지오메트리의 기본 근사가 볼록 껍질(convex hull)이기 때문입니다. 이는 시뮬레이션 엔진이 효율적으로 처리할 수 있는 근사로, 성능 좋은 시뮬레이션에 좋은 선택입니다. 아래에서 더 복잡하고 계산 비용이 높은 근사를 검토할 것입니다.

사용 중인 충돌 형태를 보려면:

  1. 뷰포트 왼쪽 상단의 눈 아이콘을 클릭합니다.

  2. Show by type > Physics > Colliders > Selected.

  3. 토러스에 녹색 선이 나타나는지 확인합니다.

이것은 충돌 형태의 디버그 뷰입니다.

Physics 디버그 메뉴를 열어 콜라이더의 솔리드 표시를 볼 수도 있습니다: 1. Window > Simulation > Debug. 2. 디버그 창에서 "Collision Mesh Debug Visualization"으로 스크롤합니다. 3. "Solid Mesh Collision Visualization"을 체크합니다. 4. 토러스를 선택하면 형태가 솔리드로 표시되는지 확인합니다.

Convex Decomposition#

약간의 비용으로 토러스 콜라이더에 볼록 형태의 구성을 통해 구멍을 만들 수 있습니다. 이 구성은 다음 방법으로 만들 수 있습니다:

  • 여러 형상을 수동으로 추가하여 생성

  • Physics Convex Decomposition으로 계산

  1. 토러스를 선택합니다.

  2. 속성 패널에서 Collision 섹션으로 스크롤하고 드롭다운에서 Convex Decomposition을 선택합니다.

  3. Advanced 탭을 열어 만족스러운 분해를 찾을 때까지 파라미터를 조정합니다.

Note

convex hull이 적을수록 일반적으로 성능이 향상됩니다.

Simulation Debug 탭에서 Explode View 거리를 늘려 콜라이더 형상을 분리하고 구성 방식을 더 잘 이해할 수 있습니다.

Collider 드롭다운에는 가장 저렴한 충돌인 Bounding Cube와 Sphere, 그리고 적합 메시 대신 구 그룹을 직접 사용하는 Convex Decomposition과 유사한 "Sphere Approximation" 모드 등 더 많은 옵션이 있습니다.

Note

삼각형 메시와 메시 단순화는 리지드 바디에서 지원되지 않고 convex hull로 폴백되지만, 서명 거리 필드(SDF)를 추가하여 리지드 바디에서 직접 삼각형 메시 지오메트리를 사용할 수 있습니다. 이를 위해 근사 드롭다운에서 SDF Mesh를 선택하세요.

리지드 바디와 콜라이더에 대한 자세한 내용은 Rigid Body Simulation과 Colliders를 확인하세요.

Contact and Rest Offset#

Collider Advanced 탭에는 특히 작고 얇은 오브젝트에서 충돌 문제가 있을 때 중요한 튜닝 파라미터 두 가지가 있습니다.

Rest Offset는 충돌 지오메트리 세트를 팽창시키거나 축소시키도록 조정할 수 있습니다. 시각적 메시가 충돌 지오메트리보다 크거나 작은 경우 충돌 위치가 시각적 표현과 일치하도록 조정하는 데 유용합니다.

Contact Offset는 Rest Offset과 무관하게 충돌 지오메트리에서 얼마나 떨어진 곳에서 시뮬레이션 엔진이 접촉 제약을 생성하기 시작하는지를 결정합니다. Contact Offset 조정의 트레이드오프는 성능 대 충돌 품질입니다. Contact Offset이 크면 많은 접촉 제약이 생성되어 계산 비용이 높아지고, 작은 Offset은 접촉이 너무 늦게 감지되어 떨림, 접촉 누락, 터널링 등의 문제가 발생할 수 있습니다(위의 CCD 참고 사항 참조).

Contacts and Friction#

충돌은 오브젝트가 서로 관통하지 않도록 하는 것 외에도 반발력(restitution)과 마찰력(friction)으로 모델링된 에너지를 전달하거나 소산시킬 수 있습니다.

접촉 모델의 파라미터는 물리 머티리얼에서 사용할 수 있습니다. Physics material을 생성하려면:

  1. Create > Physics > Physics Material로 이동합니다.

  2. Rigid Body Material을 선택합니다.

물리 머티리얼은 일반적으로 Collider Geometry에 할당되지만 그 외에는 USD 렌더 머티리얼과 유사하게 작동합니다. USD 머티리얼 해석 로직에 대한 전체 설명은 Physics Materials를 참조하세요. 예를 들어, 리지드 바디의 충돌 지오메트리마다 다른 머티리얼을 할당하거나 리지드 바디 prim에 머티리얼을 할당하여 콜라이더 자식에 설정된 모든 머티리얼을 재정의할 수 있습니다.

물리 머티리얼을 할당하려면:

  1. 콜라이더 prim을 선택합니다.

  2. Collider 설정으로 스크롤합니다.

  3. Physics Materials on Selected Models에서 원하는 머티리얼을 선택합니다. 목록은 물리 속성이 있는 머티리얼만 선택할 수 있습니다.

렌더 머티리얼에 "Add" > Physics > Rigid Body Material로 물리 머티리얼을 추가하고 렌더 머티리얼 섹션에서 머티리얼을 할당할 수도 있습니다. 물리 속성이 적용됩니다.

Compliant Contacts#

Advanced 탭에서 리지드 머티리얼이 순응(compliant, 즉 스프링-댐퍼) 접촉 역학을 생성하도록 구성할 수 있습니다. 이는 리지드 바디로 변형 가능 바디를 근사하는 데 유용할 수 있습니다.

Combine Modes#

접촉은 두 바디 간의 상호작용이므로 각 접촉 파라미터만으로는 이 상호작용이 어떻게 전개되는지 설명하기에 충분하지 않습니다. 실제 세계와 마찬가지로 한 표면의 재질 속성이 상호작용을 지배하거나 평균값으로 원활하게 결합될 수 있습니다. 이를 재현하기 위해 마찰, 반발력, 순응 접촉 댐핑은 구성 가능한 결합 모드 필드를 가지고 있습니다. 접촉의 양쪽 모두 이 결합 모드를 가지므로 결합 모드의 우선순위가 중요합니다:

드롭다운에서 낮을수록 불일치 해결에서 모드의 우선순위가 낮습니다. 따라서 average < min < multiply < max.

예를 들어 콜라이더 A의 마찰 결합 모드가 average이고 콜라이더 B가 min이면, 둘의 상호작용은 두 마찰 중 최솟값으로 해결됩니다. 결합 모드 max인 바디 C가 A와 B에 접촉하면 A와 C의 마찰은 max로, B와 C도 max로 해결됩니다.

Joints#

로봇은 일반적으로 여러 조인트 리지드 바디로 구성됩니다. 조인트는 두 바디 간에 제약을 만듭니다. 다음에서 Revolute Joint를 사용하지만 다른 조인트 유형에도 단계가 유사합니다. Joints에서 목록을 확인하세요.

조인트로 연결할 각 바디의 조인트 프레임 상대 포즈를 구성해야 합니다. 특히 조인트 프레임의 로컬 스케일링 측면에 대한 자세한 내용은 Joint Frames Section을 참조하세요.

UI를 통해 조인트를 생성할 때 조인트의 프레임은 생성을 위해 선택한 두 번째 리지드 바디의 포즈와 일치하도록 설정됩니다.

이제 다음과 같이 조인트를 만드세요:

  1. 먼저 Xform 리지드 바디를 선택하고, 그런 다음 Torus 리지드 바디를 선택합니다.

  2. Create > Physics > Joints > (Joint Type)으로 이동합니다.

이 튜토리얼에서는 Revolute Joint 유형을 사용합니다. Torus가 두 번째로 선택되었으므로 조인트가 Torus 중심에 위치합니다.

화면에 조인트의 원점과 운동 범위를 나타내는 원이 표시됩니다. 지금 시뮬레이션을 시작하면 Torus와 Cube가 함께 떨어집니다. Torus가 지면에 닿으면 큐브가 멈춥니다. 안정적인 위치에 있지만 살짝 건드리면 원형 패턴으로 내려갑니다. Shift를 누른 채 큐브를 클릭하여 상호작용해 보세요.

속성 패널을 확인하고 다음 속성을 검토합니다:

  1. Body 0: /World/XForm

  2. Body 1: /World/Torus

이들은 바디에 상대적인 포즈입니다. Position 0이 Z=-7.0임을 알 수 있습니다.

  1. Position 0: [0, 0, -7.0]

  2. Rotation 0: [0, 0, 0.0]

  3. Position 1: [0, 0, 0.0]

  4. Rotation 1: [0, 0, 0.0]

Note

아티큘레이션의 일부인 조인트를 설정할 때, Body 0가 아티큘레이션 트리 계층에서 Body 1의 부모가 되는지 확인하세요. 이렇게 하면 링크 수신 조인트 힘이나 조인트 드라이브 목표 같은 조인트 관련 양이 PhysX SDK와 USD에서 일대일 대응을 갖게 됩니다.

Joint Axis#

Revolute Joint는 하나의 자유도를 제공하며 어느 조인트 프레임 축이 자유로운지 선택할 수 있습니다. 기본적으로 X 축이 선택됩니다. Properties의 Revolute Joint 섹션에서 변경할 수 있습니다.

Joint Limits#

조인트 제한은 조인트가 원래 위치에서 얼마나 멀리 이동할 수 있는지를 결정합니다. 기본적으로 조인트 생성 시 제한이 없습니다. 조인트를 선택한 상태에서 속성 패널을 스크롤하여 Revolute Joint 섹션의 Lower Limit와 Upper Limit를 수정합니다. USD는 라디안이 아닌 도(degrees)를 사용한다는 것을 기억하세요.

Adding a Joint Drive#

Joint Drive를 사용하여 조인트가 추가한 자유도의 위치와 속도를 제어할 수 있습니다. Add Button > Physics > Angular Drive를 클릭하여 할 수 있습니다. 조인트 드라이브 구성에 대한 자세한 내용은 Tutorial 11: Tuning Joint Drive Gains을 참조하세요.

Articulation#

아티큘레이션은 조인트 바디를 위한 최적화된 시뮬레이션 구조로 로보틱스에 뛰어난 성능, 정확도, 기능을 제공합니다. 토폴로지(루프 클로징)와 조인트 지원에 관한 일부 제한이 있으며 Articulations에서 학습할 수 있습니다. 아티큘레이션 튜닝에 대한 전체 가이드는 Articulation Stability Guide를 참조하세요.

전반적인 시뮬레이션 힌트와 FAQ는 Physx Simulation Hints and FAQ를 참조하세요.

Stepping an OmniGraph with Physics#

물리 스텝이 발생하는 순간 그래프 스텝당 하나를 보장하려면 수정된 OmniGraph 버전을 사용해야 합니다.

  1. Create > Visual Scripting > Action Graph를 통해 새 Action Graph를 만듭니다.

  2. 스테이지에서 생성된 그래프를 선택하고 Raw USD Properties 섹션의 파이프라인 스테이지에서 PipelineStageOnDemand를 선택합니다.

  3. Action Graph 창에서 On Physics Step을 검색합니다. OmniGraph에 드래그 앤 드롭합니다.

  4. 평소처럼 OmniGraph를 계속 진행합니다.
