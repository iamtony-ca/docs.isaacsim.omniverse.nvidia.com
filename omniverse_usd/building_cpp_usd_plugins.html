Building C++ USD Plugins Against the Standalone Installer#

독립 실행형 Isaac Sim 설치 프로그램은 USD 런타임 공유 라이브러리(extscache/omni.usd.libs-*/bin/)와 일치하는 Python 바인딩을 포함하지만, 자체 네이티브 USD 플러그인을 컴파일하는 데 필요한 C++ 헤더나 pxrConfig.cmake 패키지 구성 파일은 포함하지 않습니다. 이 페이지는 일치하는 헤더를 얻고 Isaac Sim이 런타임에 로드하는 USD 라이브러리에 대해 네이티브 플러그인을 링크하는 지원 워크플로우를 설명합니다.

Kit C++ 익스텐션 템플릿에 대해 인트리(in-tree) 개발하는 경우, 대신 CLI extension templates에 설명된 워크플로우를 사용하세요. 아래 페이지는 설치된 Isaac Sim 패키지에 대해 독립 실행형 네이티브 USD 플러그인(런타임에 Plug::Load()를 통해 검색 및 로드되는 플러그인)을 빌드하는 데 특별히 적용됩니다.

When You Need This#

Isaac Sim 프로세스에 로드되어야 하는 C++ USD 플러그인을 빌드하고 있으며, 소스 저장소가 아닌 독립 실행형 설치 프로그램에서 시작할 때 이 워크플로우가 필요합니다.

USD는 PXR_PLUGINPATH_NAME을 탐색하고, plugInfo.json을 파싱하고, Plug::Load()를 호출하여 런타임에 플러그인을 발견합니다. Plug::Load()는 공유 라이브러리에서 정상적인 dlopen()을 수행합니다. 즉, 플러그인이 Isaac Sim 프로세스에 이미 로드된 USD 런타임 라이브러리와 ABI 호환성이 있어야 합니다.

Prerequisites#

독립 실행형 설치 프로그램 내에서 번들된 packman 런처와 Isaac Sim의 모든 외부 의존성을 고정하는 매니페스트를 찾습니다:

<isaac-sim-install>/kit/dev/tools/packman/packman
<isaac-sim-install>/kit/dev/all-deps.packman.xml

all-deps.packman.xml을 열고 usd-release 의존성 항목을 찾습니다. Isaac Sim이 컴파일된 정확한 OpenUSD 빌드를 식별하는 name과 version 속성이 있는 내부 <package> 요소가 포함됩니다. 예를 들면:

<dependency name="usd-release">
    <package name="usd.py312.manylinux_2_35_x86_64.stock.release"
             version="0.25.11.kit.2-gl.19811"/>
</dependency>

name은 플랫폼 및 Python 버전으로 인코딩됩니다. 두 속성 모두 자체 packman 프로젝트 파일에 그대로 복사하세요(아래 참조).

설치 프로그램 내에 있는 Python 개발 헤더도 필요합니다:

<isaac-sim-install>/kit/python/include/python3.XX/

USD Ar과 Tf 헤더는 TfPyObjWrapper를 통해 Boost.Python을 전이적으로 포함하므로, Python을 직접 호출하지 않는 플러그인도 #include <Python.h>를 할 수 있어야 합니다.

Pull Matching USD Dev Artifacts With Packman#

플러그인 소스 옆에 소규모 packman 프로젝트 파일을 생성합니다. 예를 들어 usd-dev.packman.xml:

<project toolsVersion="6.0">
  <!--
    Replace PACKAGE_NAME_FROM_ALL_DEPS and VERSION_FROM_ALL_DEPS with
    the inner <package> element's name and version copied verbatim
    from the usd-release entry in
    <isaac-sim-install>/kit/dev/all-deps.packman.xml.
  -->
  <dependency name="usd-release" linkPath="_deps/usd-release">
    <package name="PACKAGE_NAME_FROM_ALL_DEPS" version="VERSION_FROM_ALL_DEPS"/>
  </dependency>
</project>

그런 다음 설치 프로그램에 번들된 packman을 사용하여 일치하는 dev 아티팩트를 가져옵니다:

<isaac-sim-install>/kit/dev/tools/packman/packman pull \
    usd-dev.packman.xml --platform linux-x86_64

packman pull이 완료되면 _deps/usd-release/에 USD 개발 트리가 포함됩니다:

_deps/usd-release/
  include/
    pxr/        # OpenUSD C++ headers; in OpenUSD 25.x and later,
                # Boost.Python is vendored under pxr/external/boost/
    tbb/        # TBB headers used by USD
  lib/          # Packman-built USD libraries (do NOT link against these)
  pxrConfig.cmake
  cmake/

컴파일에는 include/를 사용합니다. pxrConfig.cmake가 드롭에 포함되지만, TBB, MaterialX, Imath 및 OpenSubdiv의 *Config.cmake 파일을 전이적으로 find_dependency() 로드하며, usd-release packman 드롭에는 이들이 포함되지 않습니다. 따라서 일반 find_package(pxr REQUIRED)가 해결에 실패합니다. 다음 섹션의 레시피는 CMake를 include/로 직접 지정하여 이를 우회합니다. _deps/usd-release/lib/ 아래의 라이브러리에 대해 플러그인을 링크하지 마세요 — 다음 섹션을 참조하세요.

Compile and Link#

빌드에는 다음이 필요합니다:

  • _deps/usd-release/include/와 <isaac-sim-install>/kit/python/include/python3.XX/의 헤더.

  • <isaac-sim-install>/extscache/omni.usd.libs-*/bin/ 링크 디렉토리(_deps/usd-release/lib/가 아닌 Isaac Sim이 런타임에 로드하는 라이브러리).

  • usd_ar, usd_sdf, usd_tf, usd_plug, usd_vt 링크 라이브러리(플러그인에 필요에 따라 더 추가).

  • 해당 라이브러리 주변의 링커 플래그 -Wl,--no-as-needed.

동일한 경로와 플래그가 모든 빌드 시스템에 적용되지만, 각 도구가 표현하는 방식은 다릅니다. 일부는 절대 파일 시스템 경로를 직접 받아들이고, 다른 일부는 사전 빌드된 라이브러리를 대상으로 래핑하거나 작업공간 로컬 경로를 통해 참조하도록 권장합니다. 빌드 시스템에 맞는 관용적인 메커니즘을 사용하세요. CMake의 예시로:

cmake_minimum_required(VERSION 3.20)
project(my_usd_plugin LANGUAGES CXX)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

# Python development headers shipped inside Isaac Sim.
set(ISAAC_SIM_INSTALL "/path/to/isaac-sim" CACHE PATH "Isaac Sim install root")
set(ISAAC_SIM_USD_LIBS "${ISAAC_SIM_INSTALL}/extscache/omni.usd.libs-<version>/bin"
    CACHE PATH "Directory containing libusd_*.so shipped with Isaac Sim")
set(ISAAC_SIM_PYTHON_INCLUDE "${ISAAC_SIM_INSTALL}/kit/python/include/python3.XX"
    CACHE PATH "Python development headers shipped inside Isaac Sim")

add_library(my_usd_plugin SHARED my_usd_plugin.cpp)

# Use the packman-pulled headers directly; do not call find_package(pxr).
# See the prose above for why pxrConfig.cmake is intentionally bypassed.
target_include_directories(my_usd_plugin PRIVATE
    "${CMAKE_SOURCE_DIR}/_deps/usd-release/include"
    "${ISAAC_SIM_PYTHON_INCLUDE}"
)

# Link against the USD libraries Isaac Sim actually loads at runtime,
# not the libraries packman pulled. Linking against packman's libs
# produces a binary that resolves to one set of pxrInternal_v0_* symbols
# at link time and a (potentially different) set at runtime, which
# manifests as undefined-symbol errors inside Plug::Load().
target_link_directories(my_usd_plugin PRIVATE "${ISAAC_SIM_USD_LIBS}")
target_link_libraries(my_usd_plugin PRIVATE
    -Wl,--no-as-needed
    usd_ar
    usd_sdf
    usd_tf
    usd_plug
    usd_vt
    -Wl,--as-needed
)

두 가지 링크 라인 세부 사항이 중요합니다:

  1. -Wl,--no-as-needed는 usd_* 라이브러리 주변에 필요합니다. 이 없이는 링커가 플러그인 클래스의 C++ vtable 및 RTTI를 통해서만 참조되는 USD 라이브러리를 삭제합니다. Plug에 의해 발견되는 모든 클래스에 대해 이는 본질적으로 의존하는 모든 USD 라이브러리입니다. 결과는 깔끔하게 링크되지만 런타임에 Plug::Load() 내에서 pxrInternal_v0_*::...에 대한 정의되지 않은 심볼 오류와 함께 실패하는 플러그인입니다.

  2. 링커에 전달되는 라이브러리 검색 경로는 <isaac-sim-install>/extscache/omni.usd.libs-*/bin/이어야 하며, _deps/usd-release/lib/가 아닙니다. packman으로 가져온 라이브러리는 동일한 OpenUSD 버전으로 빌드되지만 Isaac Sim이 포함하는 정확히 같은 바이너리가 아닐 수 있으며, 빌드 플래그나 심볼 버전 지정의 불일치는 무음 ABI 손상을 유발합니다.

런타임에 PXR_PLUGINPATH_NAME을 설정하거나(또는 USD가 이미 스캔하는 디렉토리 아래에 plugInfo.json을 배치하여) Isaac Sim이 스테이지 열기 중에 플러그인을 발견하도록 합니다.

Plugin Source Requirements#

USD의 플러그인 등록 매크로(TF_REGISTRY_FUNCTION, TF_REGISTRY_FUNCTION_WITH_TAG)는 전역 네임스페이스 범위에서 확장되며 자격 없는 Tf_RegistryInit와 라이브러리당 태그 MFB_ALT_PACKAGE_NAME을 참조합니다. 두 가지 소스 수준 요구 사항은 선택 사항이 아닙니다:

  1. 플러그인 대상당 컴파일 시간에 세 가지 전처리기 매크로를 정의합니다. 이들은 플러그인을 USD의 레지스트리에 식별합니다. 값은 관례적으로 세 가지 대소문자 변형의 동일한 짧은 이름입니다. CMake에서는 다음과 같습니다:

    target_compile_definitions(my_usd_plugin PRIVATE
    MFB_PACKAGE_NAME=myUsdPlugin
    MFB_ALT_PACKAGE_NAME=myUsdPlugin
    MFB_PACKAGE_MODULE=MyUsdPlugin
)

  2. TF_REGISTRY_FUNCTION을 사용하는 모든 번역 단위 상단 근처에 PXR_NAMESPACE_USING_DIRECTIVE를 배치하세요. 이는 매크로 확장 내의 자격 없는 Tf_RegistryInit가 OpenUSD 인라인 네임스페이스로 해결되도록 합니다:

    #include <pxr/pxr.h>
#include <pxr/base/tf/type.h>
#include <pxr/usd/ar/resolver.h>

PXR_NAMESPACE_USING_DIRECTIVE

class MyResolver : public ArResolver { /* ... */ };

TF_REGISTRY_FUNCTION(TfType) {
    TfType::Define<MyResolver, TfType::Bases<ArResolver>>();
}

둘 중 하나가 없으면 플러그인이 컴파일되지 않습니다.

Keeping Headers and Runtime In Sync#

OpenUSD는 pxrInternal_v0_* 인라인 네임스페이스에 릴리스 버전을 인코딩합니다. 하나의 OpenUSD 릴리스에 대해 컴파일된 플러그인은 다른 OpenUSD 릴리스를 링크하는 프로세스에 로드될 수 없습니다. 맹글링된 심볼 이름이 일치하지 않아 Plug::Load() 내에서 정의되지 않은 심볼 오류가 발생합니다.

Isaac Sim에 포함된 OpenUSD 버전은 릴리스 간에 변경될 수 있습니다. 아래 표는 알려진 매핑을 기록합니다. 항상 대상으로 하는 설치 프로그램의 all-deps.packman.xml 내의 usd-release 항목을 검사하여 정확한 버전을 확인하세요.

Isaac Sim 버전

OpenUSD 버전

인라인 네임스페이스 (pxrInternal_v0_*)

5.1.0

24.05

pxrInternal_v0_24_5__pxrReserved__

6.0.0

25.11

pxrInternal_v0_25_11__pxrReserved__

Caution

이 페이지의 packman 접근 방식은 대상으로 하는 설치 프로그램의 일치하는 OpenUSD 헤더를 자동으로 가져오므로 인라인 네임스페이스 문자열을 직접 읽을 필요가 없어야 합니다. 이것들은 진단 보조로만 여기에 나열됩니다: pxrInternal_v0_24_5를 포함하는 정의되지 않은 심볼이 보이지만 플러그인이 pxrInternal_v0_25_11에 대해 컴파일된 경우, 헤더와 런타임이 동기화되지 않았으며 링크 단계가 잘못된 USD 라이브러리를 가리키고 있는 것입니다. 이 표에 의존하기 전에 nm -D --defined-only extscache/omni.usd.libs-*/bin/libusd_tf.so | head로 설치 프로그램의 정확한 네임스페이스 문자열을 확인하세요.
