Walkthrough: UR10 Bin Stacking#

Warning

[DEPRECATED]: Cortex 프레임워크는 Isaac Sim 6.0.0부터 지원이 중단되었으며 향후 릴리스에서 제거될 예정입니다. 동작 프로그래밍을 위해 동작 트리에는 py_trees, 유한 상태 기계에는 transitions와 같은 오픈 소스 라이브러리로 마이그레이션하세요. Isaac Sim 7.0에는 이러한 라이브러리를 사용하는 예제가 포함될 예정입니다.

이 튜토리얼은 UR10 로봇을 사용한 완전한 빈 쌓기 응용 프로그램을 단계별로 설명합니다. 이 예제에서는 컨베이어 벨트, 빈을 쌓아야 할 팔레트, 흡착 그리퍼가 있는 UR10 로봇이 포함된 미리 설계된 USD 환경을 사용합니다. 응용 프로그램은 장면에 추가하지 않고(보이지 않는 충돌 장애물 제외) 미리 설계된 USD 환경의 기존 USD 요소를 제어합니다.

모든 명령줄 예제에서 Isaac Sim 파이썬 스크립트(<isaac_sim_root>/python.sh Linux, <isaac_sim_root>\python.bat Windows)의 약어로 isaac_python을 사용합니다. 명령줄은 작업 디렉토리 standalone_examples/deprecated/api/isaacsim.cortex.framework에 상대적으로 작성됩니다.

다음 데모를 실행하세요

isaac_python demo_ur10_conveyor_main.py

Isaac Sim이 시작되면 재생을 누르세요. 빈 쌓기 데모가 실행되는 것을 볼 수 있습니다.

이 튜토리얼은 데모를 단계별로 설명합니다.

설정은 흡착 그리퍼가 있는 UR10 로봇이 컨베이어에서 팔레트로 빈을 이동하는 것입니다. 빈은 뒤집어서 쌓아야 하므로 올바른 방향으로 오는 빈은 쌓기 전에 뒤집기 스테이션에서 뒤집힙니다.

이 데모는 최상위 디스패치 노드가 여러 순차적 상태 기계 중에서 선택하는 얕은 결정자 네트워크를 사용합니다.

개별 동작은 모션 동작을 형성하기 위해 장애물 영역이 전략적으로 자동으로 켜지고 꺼지는 일반적인 RMPflow 프로그래밍 기법을 보여줍니다. 이 장애물 영역은 장면 USD에서 모델링되지만 기본적으로 보이지 않습니다. 가시성 토글을 시도해보세요. World/Ur10Table/Obstacles로 이동하여 FlipStationSphere, NavigationDome, NavigationBarrier, NavigationFlipStation의 가시성을 토글하여 확인하세요.

빈 배치 동작은 또한 접근 방향 매개변수와 함께 RMPflow의 반응성을 활용하여 빈 사이의 잘못된 정렬을 즉석에서 수정하는 자동 조정 동작을 만듭니다.

Top-level dispatch#

결정자 네트워크의 진입점은 구성에서 볼 수 있듯이 Dispatch 노드입니다. DfNetwork는 결정자 네트워크 구조입니다. 루트(Dispatch 노드)와 모든 결정자/상태 노드 내에서 멤버로 사용 가능한 컨텍스트 객체가 전달됩니다:

def make_decider_network(robot):
    return DfNetwork(Dispatch(), context=BinStackingContext(robot))

컨텍스트 객체는 각 노드에 로봇의 명령 API와 모니터에 의해 추출된 논리적 상태에 대한 접근을 제공합니다. Dispatch 노드의 decide() 로직은 논리적 상태를 고려하면 꽤 간단합니다.

class Dispatch(DfDecider):
    def __init__(self):
        super().__init__()

        self.add_child("flip_bin", FlipBin())
        self.add_child("pick_bin", PickBin())
        self.add_child("place_bin", PlaceBin())
        self.add_child("go_home", make_go_home())
        self.add_child("do_nothing", DfStateMachineDecider(DoNothing()))

    def decide(self):
        if self.context.stack_complete:
            return DfDecision("go_home")

        if self.context.has_active_bin:
            if not self.context.active_bin.is_attached:
                return DfDecision("pick_bin")
            elif self.context.active_bin.needs_flip:
                return DfDecision("flip_bin")
            else:
                return DfDecision("place_bin")
        else:
            return DfDecision("go_home")

논리적 상태에는 다음이 포함됩니다:

  • stack_complete: 모든 빈이 팔레트에 있는지 여부를 기록합니다.

  • active_bin: 현재 사용 중인 빈. 빈은 스택에 배치될 때까지 활성 상태를 유지합니다. 그런 다음 컨베이어 끝에 있는 빈 중에서 새 빈이 선택됩니다.

  • active_bin.is_attached: 활성 빈이 흡착 그리퍼를 통해 엔드 이펙터에 부착되어 있는지 나타냅니다.

  • active_bin.needs_flip: 엔드 이펙터에 부착된 빈이 올바른 방향(뒤집기 필요) 또는 뒤집힌 방향(뒤집기 불필요)인지 나타냅니다.

결정 로직은 단순해집니다: 스택이 완료되었거나 활성 빈이 없으면 홈으로 이동합니다. 그렇지 않고 활성 빈이 있으면, 이미 그리퍼에 없으면 빈을 집고 뒤집어야 하면 뒤집습니다. 그런 다음 빈을 스택에 놓습니다. 결정자 네트워크는 이 결정 로직을 읽기 쉬운 형태로 작성하기 쉽게 만듭니다.

Sequential state machines#

집기, 뒤집기, 놓기 동작을 구현하는 순차적 상태 기계는 각각 구조가 유사합니다.

class PickBin(DfStateMachineDecider):
    def __init__(self):
        super().__init__(
            DfStateSequence(
                [
                    ReachToPick(),
                    DfWaitState(wait_time=0.5),
                    DfSetLockState(set_locked_to=True, decider=self),
                    CloseSuctionGripper(),
                    DfTimedDeciderState(DfLift(0.3), activity_duration=0.4),
                    DfSetLockState(set_locked_to=False, decider=self),
                ],
            )
        )


class FlipBin(DfStateMachineDecider):
    def __init__(self):
        super().__init__(
            DfStateSequence(
                [
                    LiftAndTurn(),
                    MoveToFlipStation(),
                    DfSetLockState(set_locked_to=True, decider=self),
                    OpenSuctionGripper(),
                    ReleaseFlipStationBin(duration=0.65),
                    DfSetLockState(set_locked_to=False, decider=self),
                ]
            )
        )


class PlaceBin(DfStateMachineDecider):
    def __init__(self):
        super().__init__(
            DfStateSequence(
                [
                    ReachToPlace(),
                    DfWaitState(wait_time=0.5),
                    DfSetLockState(set_locked_to=True, decider=self),
                    OpenSuctionGripper(),
                    DfTimedDeciderState(DfLift(0.1), activity_duration=0.25),
                    DfWriteContextState(lambda ctx: ctx.mark_active_bin_as_complete()),
                    DfSetLockState(set_locked_to=False, decider=self),
                ],
            )
        )

주목할 만한 기능은 결정자 네트워크의 잠금 및 잠금 해제 사용입니다. 결정자 네트워크는 본질적으로 반응형이므로 선점되어서는 안 되는 원자적 상태 기계 동작은 명시적으로 잠겨야 합니다. 순차적 상태 기계는 결정자 네트워크를 잠그고 해제하기 위해 DfSetLockState를 사용합니다. 또한 PlaceBin은 배치 절차를 수행한 후 활성 빈을 완료로 표시하는 컨텍스트 함수를 호출하기 위해 DfWriteContextState를 사용합니다.

Navigation obstacle monitors#

MotionCommander의 기본 모션 생성기는 등록된 장애물을 자동으로 피하는 RMPflow입니다. 그러나 상호 작용을 가능하게 하기 위해 이러한 장애물을 꺼야 할 때가 있습니다. 예를 들어 이동 중에는 조작 가능한 물체를 장애물로 피하고 싶지만 도착하면 잡아야 합니다. isaacsim.cortex.framework/isaacsim/cortex/framework/obstacle_monitor_context.py의 ObstacleMonitor와 ObstacleMonitorContext 클래스는 프로그래밍된 조건에 따라 장애물을 자동으로 켜고 끄는 장애물 모니터 개발을 용이하게 합니다.

위에 나열된 파일에서 ObstacleMonitor 구현을 살펴보세요. 생성 시 모니터링할 장애물 세트와 컨텍스트 객체를 받으며, API는 파생 클래스가 컨텍스트 객체에서 접근 가능한 정보를 기반으로 장애물을 언제 활성화하고 비활성화해야 하는지 정의하는 is_obstacle_required()를 구현하도록 요구합니다. 메서드는 결정자/상태 객체와 유사하게 self.context로 컨텍스트에 접근합니다. API는 또한 모니터를 활성화하고 비활성화하기 위해 activate_autotoggle()과 deactivate_autotoggle()을 제공합니다. 활성 상태일 때 is_obstacle_required()의 참/거짓 값에 따라 자동으로 장애물을 활성화하고 비활성화합니다. 비활성화 시 장애물이 비활성화되고 모니터가 재활성화될 때까지 비활성화 상태를 유지합니다.

ObstacleMonitorContext는 자동으로 monitor_obstacles 논리적 상태 모니터를 추가하는 편리한 기본 클래스로, 파생 클래스는 add_obstacle_monitors()를 사용하여 장애물 모니터 객체만 추가하면 됩니다.

BinStackingContext 객체는 ObstacleMonitorContext에서 파생되어 팔레트와 컨베이어 사이의 네비게이션 동작과 빈을 뒤집는 동안 빈 뒤집기 스테이션 주변 네비게이션을 형성하는 데 사용하는 두 개의 장애물 모니터를 추가합니다. BinStackingContext 생성자에서 구성하고 추가합니다:

class BinStackingContext(ObstacleMonitorContext):
    def __init__(self, robot):
        super().__init__()

        ...

        self.flip_station_obs_monitor = FlipStationObstacleMonitor(self)
        self.navigation_obs_monitor = NavigationObstacleMonitor(self)
        self.add_obstacle_monitors([self.flip_station_obs_monitor, self.navigation_obs_monitor])

두 모니터 모두 간단한 로직을 가집니다:

class FlipStationObstacleMonitor(ObstacleMonitor):
    def __init__(self, context):
        super().__init__(context, [context.world.scene.get_object("flip_station_sphere")])

    def is_obstacle_required(self):
        eff_T = self.context.robot.arm.get_fk_T()
        eff_R, eff_p = math_util.unpack_T(eff_T)
        eff_ax, _, _ = math_util.unpack_R(eff_R)

        grasp_p = self.context.active_bin.grasp_T[:3, 3]
        grasp_ax = self.context.active_bin.grasp_T[:3, 0]
        v = eff_p - grasp_p
        dist = v.dot(grasp_ax)
        orth_dist = np.linalg.norm(v - dist * grasp_ax)
        return not (dist < 0.02 and grasp_ax.dot(eff_ax) > 0.75 and orth_dist < 0.03)


class NavigationObstacleMonitor(ObstacleMonitor):
    def __init__(self, context):
        obstacles = [
            context.world.scene.get_object("navigation_dome_obs"),
            context.world.scene.get_object("navigation_barrier_obs"),
            context.world.scene.get_object("navigation_flip_station_obs"),
        ]
        super().__init__(context, obstacles)

    def is_obstacle_required(self):
        target_p, _ = self.context.robot.arm.target_prim.get_world_pose()

        ref_p = np.array([0.6, 0.37, -0.99])
        eff_p = self.context.robot.arm.get_fk_p()

        ref_p[2] = 0.0
        eff_p[2] = 0.0
        target_p[2] = 0.0

        s_target = np.sign(np.cross(target_p, ref_p)[2])
        s_eff = np.sign(np.cross(eff_p, ref_p)[2])
        is_required = s_target * s_eff < 0.0
        return is_required

FlipStationObstacleMonitor는 뒤집기 스테이션 주변의 구형 장애물인 flip_station_sphere를 모니터링합니다. 활성 상태일 때 엔드 이펙터가 포즈 목표를 향해 모션 명령의 접근 방향을 따라 하강할 때까지 장애물을 활성화합니다. 모니터는 빈을 아래에서 해제하고 위에서 집기 위해 이동한 후 뒤집기 스테이션과 빈(스테이션에 놓인)을 피하는 데 사용됩니다. ReachToPick 클래스(빈이 뒤집기 스테이션에 있는지 여부에 관계없이 빈을 집어야 할 때마다 사용됨)에 진입 시 활성화되고 종료 시 비활성화됩니다.

class ReachToPick(MoveWithNavObs):
    ...

    def enter(self):
        super().enter()
        self.context.flip_station_obs_monitor.activate_autotoggle()

    def step(self): ...

    def exit(self):
        super().exit()
        self.context.flip_station_obs_monitor.deactivate_autotoggle()

마찬가지로 NavigationObstacleMonitor는 이동 중에 로봇 베이스와 현재 빈 스택을 피하기 위해 팔레트와 컨베이어 사이(양방향)의 네비게이션 동작을 형성하는 장애물 컬렉션을 모니터링합니다. 하나에서 다른 하나로 이동하는 동안에는 필요하지만, 팔이 목적지 영역(팔레트 또는 컨베이어)에 도달하면 더 이상 필요하지 않습니다.

MoveWithNavObs 상태 객체는 네비게이션 장애물을 자동으로 토글하는 진입 및 종료 조건으로 Move 상태를 확장합니다:

class MoveWithNavObs(Move):
    def enter(self):
        super().enter()
        self.context.navigation_obs_monitor.activate_autotoggle()

    def exit(self):
        super().exit()
        self.context.navigation_obs_monitor.deactivate_autotoggle()

이 클래스는 위에 나열된 PickBin과 PlaceBin에서 사용하는 ReachToPick 상태와 ReachToPlace 상태 모두의 기본 클래스입니다.

Robustness reactivity on placement#

엔드 이펙터에 빈 부착과 빈의 쌓기 정렬이 모두 물리적으로 시뮬레이션됩니다. 오류를 조정하지 않고 맹목적으로 빈을 잡고 목표로 이동하면 서로 제대로 접하지 않는 약간 잘못 정렬된 빈이 됩니다. 반응형 모션 생성기(RMPflow)를 사용하므로 반응형 조정 구현은 간단합니다. ReachToPlace에서 조정은 step() 메서드에서 매 사이클 이루어집니다:

class ReachToPlace(MoveWithNavObs):
    ...

    def step(self):
        if self.bin_under is not None:
            bin_under_p, _ = self.bin_under.bin_obj.get_world_pose()
            bin_grasped_p, _ = self.context.active_bin.bin_obj.get_world_pose()
            xy_err = bin_under_p[:2] - bin_grasped_p[:2]
            if np.linalg.norm(xy_err) < 0.02:
                self.target_p[:2] += 0.1 * (bin_under_p[:2] - bin_grasped_p[:2])

        target_pose = PosePq(self.target_p, math_util.matrix_to_quat(self.target_R))

        approach_params = ApproachParams(direction=0.15 * np.array([0.0, 0.0, -1.0]), std_dev=0.005)
        posture_config = self.context.robot.default_config
        self.update_command(
            MotionCommand(target_pose=target_pose, approach_params=approach_params, posture_config=posture_config)
        )

빈을 다른 빈 위에 놓는 경우(bin_under) 이 코드는 그리퍼의 빈과 그 아래 빈 사이의 xy 위치 정렬 오류를 기반으로 엔드 이펙터 목표를 조정합니다. (방향 정렬은 일반적으로 성공적인 배치에 이미 충분합니다.)

또한 RMPflow는 시뮬레이터에서 반응형 상태 피드백을 받도록 구성되어 있으며, 목표 도달을 위해 타이트한 접근 매개변수(std_dev=0.005)를 사용하므로 접근 시 좁은 깔때기를 따라야 합니다. 첫 번째 접근 시 빈이 잘못 정렬되면 빈 물리가 엔드 이펙터를 그 깔때기에서 밀어내고 다시 접근을 시도합니다.

이 조합은 로봇이 빈의 위치를 반응적으로 조정하고 올바르게 될 때까지 접근을 재시도(필요시 반복적으로)하게 합니다. 조정 과정이 충분한 경우가 많지만 주기적으로 접근을 재시도해야 합니다. 일반적으로 단 한 번의 재시도로 충분합니다. 이는 미묘한 동작이고 코드는 간결하지만, 약 85%의 성공적인 빈 배치와 100% 성공의 차이입니다.

Logical state context#

BinStackingContext 객체는 또한 위의 동작을 지원하는 데 필요한 모든 논리적 상태를 모니터링합니다. 이 모니터들은 생성자에서 설정되고 논리적 상태는 reset() 메서드에서 재설정/초기화됩니다:

class BinStackingContext(ObstacleMonitorContext):
    def __init__(self, robot):
        super().__init__()
        ...

        self.add_monitors(
            [
                BinStackingContext.monitor_bins,
                BinStackingContext.monitor_active_bin,
                BinStackingContext.monitor_active_bin_grasp_T,
                BinStackingContext.monitor_active_bin_grasp_reached,
                self.diagnostics_monitor.monitor,
            ]
        )

        def reset(self):
            super().reset()

            # Find the collection of bins in the world scene.
            self.bins = []
            i = 0
            while True:
                name = "bin_{}".format(i)
                bin_obj = self.world.scene.get_object(name)
                if bin_obj is None:
                    break
                self.bins.append(BinState(bin_obj))
                i += 1

            self.active_bin = None
            self.stacked_bins.clear()

이 모니터들은 다음을 수행합니다:

  1. Monitor bins: 활성 빈이 없으면 컨베이어 끝에 빈이 있는지 확인하고 있으면 활성화합니다.

  2. Monitor the active bin: 바닥에 떨어진 빈을 비활성화합니다.

  3. Monitor the grasp transform of the active bin: 현재 활성 빈의 최적 그립을 모니터링합니다.

  4. Monitor whether the active bin is reached: 엔드 이펙터와 원하는 그립 변환 사이의 근접성과 흡착 그리퍼가 "닫혀" 있는지 여부를 기반으로 active_bin.{is_grasp_reached,is_attached} 플래그를 설정합니다.

  5. Monitor diagnostics: 읽기 쉬운(조절된) 속도로 논리적 상태에 대한 일부 정보를 출력합니다.
