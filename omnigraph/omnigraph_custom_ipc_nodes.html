Building Custom IPC OmniGraph Nodes#

이 가이드는 Isaac Sim에서 프로세스 간 통신(IPC)을 위한 OmniGraph 노드를 구성하는 방법을 설명합니다. 노드 스키마, BaseResetNode를 사용한 전송 수명 주기, compute 내 비차단 I/O, 그리고 전송 라이브러리를 의존성으로 추가하는 방법을 다룹니다. OmniGraph 패턴은 사용하는 IPC 스택에 관계없이 적용됩니다. 작동 예제는 isaacsim.examples.ipc로, C++ 및 Python에서 BSD 소켓을 통한 클록 전송 및 스텝 수신 노드 쌍입니다. 튜토리얼은 CLI 템플릿으로 새 익스텐션의 스캐폴딩으로 시작하여 IPC 코드를 작성하기 전에 작동하는 빌드 스켈레톤을 갖추게 합니다.

Note

이 워크플로우는 Isaac Sim 저장소의 소스 체크아웃이 필요합니다. pip 패키지나 바이너리 릴리스에서는 지원되지 않습니다. 시작 전에 GitHub 저장소를 클론하세요.

Note

이 튜토리얼의 모든 명령은 build.sh/build.bat와 repo.sh/repo.bat가 있는 디렉토리인 Isaac Sim 저장소 루트에서 실행됩니다.

Before You Start#

Prerequisites:

  • Custom C++ extensions — Kit C++ Extension Template.

  • OmniGraph — OmniGraph Core Concepts 및 Basic OmniGraph Tutorial.

  • Custom nodes — Custom Python Nodes 및 Custom C++ Nodes.

선택: Isaac Sim OmniGraph Tutorial — Action Graph 에디터를 처음 사용하는 경우.

참고

완전한 Python 노드 예제는 ROS 2 Python Custom OmniGraph Node를 참조하세요. ROS 2 컨텍스트이지만, internal_state() 팩토리, BaseResetNode, db.per_instance_state, og.ExecutionAttributeState.ENABLED 패턴은 모든 IPC 노드에서 동일합니다.

Scaffold Your Extension#

IPC 코드를 작성하기 전에 CLI 템플릿으로 익스텐션 스켈레톤을 생성합니다:

./repo.sh template new

.\repo.bat template new

메시지가 표시되면 Isaac Sim OmniGraph Node Extension을 선택합니다. 다음을 입력하라는 메시지가 표시됩니다:

  • extension_name 점으로 구분된 식별자, 예: isaacsim.my.ipc.nodes.

  • title Extensions 창에 표시되는 사람이 읽을 수 있는 이름.

  • description 간단한 요약.

  • category Action Graph 노드 라이브러리에서 노드를 그룹화하는 데 사용됩니다(예: Simulation).

템플릿은 source/extensions/<extension_name>/ 아래에 전체 익스텐션 스켈레톤을 생성합니다:

source/extensions/<extension_name>/
├── config/extension.toml          ← metadata, dependencies, test entries
├── nodes/
│   ├── OgnExampleCpp.ogn          ← rename/replace with your IPC node schema
│   └── OgnExampleCpp.cpp          ← rename/replace with your IPC node implementation
├── plugins/<extension_name>/
│   └── PluginInterface.cpp        ← Carbonite plugin + OGN registration (keep as-is)
├── bindings/<extension_name>/
│   └── Bindings.cpp               ← pybind11 bindings; acquires the Carbonite interface (keep as-is)
├── include/<extension_name>/
│   └── IExampleNodes.h            ← Carbonite interface (keep as-is)
├── python/nodes/
│   ├── OgnExamplePython.ogn       ← rename/replace with your Python node schema
│   └── OgnExamplePython.py        ← rename/replace with your Python Node Implementation
├── python/impl/extension.py       ← calls acquire_example_ipc_interface() on startup to load the plugin
├── python/tests/                  ← test modules go here
└── premake5.lua                   ← build configuration

빌드 단계는 Isaac Sim에서 익스텐션이 로드되기 전에 필요합니다 — C++ 플러그인을 컴파일하고 노드를 등록하는 OmniGraph 데이터베이스 파일을 생성합니다. 지금 실행합니다:

./build.sh

.\build.bat

생성된 노드 — Example C++ Node(OgnExampleCpp)와 Example Python Node(OgnExamplePython) — 는 입력 값을 두 배로 만드는 플레이스홀더 스텁입니다. Action Graph 라이브러리의 표시 이름은 파일 이름이 아닌 각 .ogn 파일의 uiName 필드에서 가져옵니다. 아래 섹션을 진행하면서 실제 IPC 노드로 이름을 바꾸거나 교체합니다.

시도하기: 스캐폴드 확인

위 빌드가 완료된 후 스캐폴드가 플레이스홀더 노드를 등록하는지 확인합니다:

  1. 별도로 설치된 Isaac Sim이 아닌 리포지토리 빌드된 Isaac Sim을 실행합니다:

    ./_build/linux-x86_64/release/isaac-sim.sh

    .\_build\windows-x86_64\release\isaac-sim.bat

  2. Window → Extensions를 열고 익스텐션 이름(예: isaacsim.my.ipc.nodes)을 검색하여 활성화합니다.

  3. Window → Graph Editors → Action Graph를 열고 노드 라이브러리에서 Example C++ Node와 Example Python Node를 검색합니다. 이 이름은 스캐폴드의 .ogn 파일의 uiName 필드와 일치합니다.

두 노드가 모두 나타나면 스캐폴드가 올바르게 구성된 것입니다. Design and Implement Your Nodes로 진행하여 플레이스홀더를 교체합니다.

노드가 나타나지 않으면 다음을 확인합니다:

  • 빌드(./build.sh on Linux, .\build.bat on Windows)가 오류 없이 완료되었는지 확인합니다. 노드는 C++ 플러그인이 컴파일되고 빌드에서 OmniGraph 데이터베이스 파일이 생성될 때까지 등록되지 않습니다.

  • 리포지토리 빌드된 Isaac Sim(./_build/linux-x86_64/release/isaac-sim.sh 또는 .\_build\windows-x86_64\release\isaac-sim.bat)을 실행했는지 확인합니다. 별도로 설치된 Isaac Sim은 저장소 exts/ 디렉토리를 검색하지 않으므로 익스텐션을 찾지 못합니다.

  • Window → Extensions에서 익스텐션이 활성화(녹색 토글)되어 있는지 확인합니다. 빌드 전에 활성화되어 있었다면 비활성화 후 다시 활성화합니다.

  • .ogn 스키마 또는 코드 변경 후에는 빌드를 다시 실행하고 Isaac Sim을 재시작해야 업데이트된 노드가 표시됩니다.

Add Your Transport Library#

노드 코드를 작성하기 전에 IPC와 직렬화를 제공하는 라이브러리를 연결합니다. 생성된 config/extension.toml과 premake5.lua가 이미 준비되어 있습니다. 아래 섹션에서 항목을 추가할 위치를 보여줍니다.

Python#

Isaac Sim은 NumPy 및 SciPy를 포함한 많은 일반 패키지를 사전 번들링한 pip 아카이브(omni.isaac.core_archive 및 관련 익스텐션)를 제공합니다. 라이브러리가 이미 해당 아카이브에 있으면 추가 설정 없이 직접 임포트할 수 있습니다.

패키지가 번들링되어 있지 않은 경우 config/extension.toml에 선언합니다. 예:

[python.pipapi]
requirements = ["pyzmq>=25", "grpcio"]  # replace with your actual packages
use_online_index = true

Isaac Sim은 이것을 익스텐션 시작 시 해석합니다. use_online_index = true가 설정되어 있어야 합니다. 생략하거나 false로 설정하면 omni.kit.pipapi가 경고를 기록하고 requirements 목록 전체를 건너뜁니다.

C++#

사전 빌드된 네이티브 라이브러리는 packman을 통해 처리됩니다. 이 단계는 Kit Extension C++ template documentation에 설명된 것과 동일한 패턴을 따릅니다:

  1. 의존성 선언. deps/ext-deps.packman.xml에 라이브러리를 추가합니다. 이것은 익스텐션별 의존성을 위한 지정된 파일입니다(Kit SDK deps와 별도). 압축 해제된 트리는 일반적으로 _build/target-deps/<libname>/ 아래에 위치합니다:

    <project toolsVersion="5.0">
  <dependency name="mylib" linkPath="../_build/target-deps/mylib">
    <package name="mylib" version="1.2.3" />
  </dependency>
</project>

  2. ``premake5.lua`` 업데이트. 포함 및 라이브러리 디렉토리를 가리키고 링크를 추가합니다:

    includedirs { "%{target_deps}/mylib/include" }
libdirs     { "%{target_deps}/mylib/lib/%{platform}" }
links       { "mylib" }

  3. 런타임 공유 라이브러리. 라이브러리가 .so/.dll로 제공되는 경우, 익스텐션 플러그인 옆에 번들하거나 extension.toml의 [native.library] 아래에 나열하여 Kit의 로더가 찾을 수 있도록 합니다.

샘플 익스텐션은 표준 BSD 소켓 API만 사용하며 플러그인 자체 외에 추가 네이티브 라이브러리 항목이 없습니다.

Design and Implement Your Nodes#

Design Principle#

IPC 노드를 얇게 유지합니다. 직렬화와 전송만 처리해야 합니다. 시뮬레이션 데이터 읽기(관절 위치, 센서 데이터, 시뮬레이션 시간)는 IPC 노드 앞에 그래프에 연결된 상위 빌트인 노드에 속합니다. 하위 처리 또는 명령 쓰기는 그 후의 다른 노드에 속합니다. 이렇게 하면 compute가 빠르고 그래프 레이아웃이 자체 문서화됩니다.

What Every IPC Node Requires#

모든 커스텀 IPC 노드는 전송에 관계없이 동일한 여섯 가지가 필요합니다:

  • 노드 스키마(.ogn 파일) — 입력(URI, 설정), 출력(데이터, execOut), 상태를 선언합니다. isaacsim.examples.ipc의 nodes/ 아래 샘플 .ogn 파일을 참조로 사용합니다.

  • BaseResetNode 서브클래스 — 인스턴스별 상태(소켓, 버퍼, 핸들)를 보유합니다. 시뮬레이션이 중지되거나 입력이 변경될 때 전송을 해제하도록 reset()(C++) 또는 custom_reset()(Python)을 구현합니다.

  • 수명 주기 분할이 있는 compute(db):

    • 입력 변경(URI, 설정) 감지 → reset 및 해제 호출

    • 준비되지 않은 경우 전송 열기 시도 → 실패 시 조기 반환(다음 평가에서 재시도)

    • 비차단 I/O 수행(전송 또는 수신 시도)

    • db.outputs 쓰기 및 execOut 발동

  • 비차단 I/O — compute에서 무기한 차단하지 않습니다. 수신 시도, 타임아웃, 또는 느린 경로를 워커 스레드로 오프로드합니다(Performance Considerations 참조).

  • compute 끝에 execOut를 발동하여 전송 작업이 완료되었거나 새 데이터가 준비되었음을 하위 노드에 신호합니다. 언제 발동할지는 사용자가 제어합니다. 예를 들어 매 평가마다, 성공적인 전송 시에만, 또는 전체 메시지가 수신될 때만 발동합니다.

  • 전송 라이브러리 — 의존성으로 추가하고(Add Your Transport Library 참조) TCP 헬퍼를 스택의 API로 교체합니다.

OGN Schema Quick Reference#

각 .ogn 파일은 노드의 등록된 타입 이름을 키로 하는 단일 JSON 객체입니다. Python IPC 노드의 최소 스키마는 다음과 같습니다:

{
    "MyNodeName": {
        "version": 1,
        "language": "Python",
        "description": "One-line description shown in the node library.",
        "metadata": { "uiName": "My Node Display Name" },
        "categoryDefinitions": "config/CategoryDefinition.json",
        "categories": "myCategory",
        "inputs": {
            "execIn":  { "type": "execution", "description": "Trigger." },
            "uri":     { "type": "string",    "description": "...", "default": "tcp://127.0.0.1:5550" },
            "myValue": { "type": "double",    "description": "...", "default": 0.0 }
        },
        "outputs": {
            "execOut":  { "type": "execution", "description": "Output execution port." },
            "myTokens": { "type": "token[]",   "description": "Array of token outputs." }
        }
    }
}

일반 스칼라 타입: "string", "double", "float", "int", "uint", "bool", "execution". 배열 변형은 []를 추가합니다: "double[]", "float[]", "token[]" 등. "default" 키는 비실행 스칼라 입력에 필요하며, 배열 입력에는 []를 사용합니다.

categoryDefinitions는 nodes/ 디렉토리에 상대적인 경로로, 카테고리 키를 사람이 읽을 수 있는 표시 문자열로 매핑하는 JSON 파일을 가리킵니다:

{
    "categoryDefinitions": {
        "myCategory": "My node group label in the Action Graph library"
    }
}

C++ Node Implementation#

BaseResetNode는 isaacsim.core.includes에 선언됩니다. 이 익스텐션은 컴파일 타임 전용 의존성으로, extension.toml의 [dependencies]에 추가하지 마세요. 대신 premake5.lua에 헤더 경로를 추가합니다:

includedirs { "%{root}/source/extensions/isaacsim.core.includes/include" }

그런 다음 .cpp 파일에 헤더를 포함합니다:

#include <isaacsim/core/includes/BaseResetNode.h>

인스턴스별 노드 클래스를 isaacsim::core::includes::BaseResetNode에서 파생시킵니다. 해당 베이스는 타임라인 정지 이벤트를 구독하고 시뮬레이션이 멈춘 후 전송 핸들이 열린 상태로 남지 않도록 reset()을 호출합니다.

생성된 OgnExampleCpp 스텁을 다음과 같은 클래스로 교체합니다(TCP 구현은 isaacsim.examples.ipc의 OgnSimpleSendSimulationClockCpp.cpp 참조):

#include <isaacsim/core/includes/BaseResetNode.h>

#include <OgnMyIpcNodeCppDatabase.h>
#include <memory>
#include <string>

using isaacsim::core::includes::BaseResetNode;

class OgnMyIpcNodeCpp : public BaseResetNode
{
public:
    static bool compute(OgnMyIpcNodeCppDatabase& db)
    {
        auto& state = db.perInstanceState<OgnMyIpcNodeCpp>();

        // Detect input changes (e.g. URI) and reset transport.
        const std::string uriIn(db.inputs.uri());
        if (state.m_handle && state.m_handle->getUri() != uriIn)
        {
            state.reset();
        }

        const bool success = state.ensureOpenAndTransfer(db);
        // Fire execOut unconditionally (send nodes). For receive nodes, fire only when
        // a complete message arrives (i.e. only when success == true).
        db.outputs.execOut() = omni::graph::core::kExecutionAttributeStateEnabled;
        return success;
    }

    static void releaseInstance(NodeObj const& nodeObj, GraphInstanceID instanceId)
    {
        auto& state = OgnMyIpcNodeCppDatabase::sPerInstanceState<OgnMyIpcNodeCpp>(nodeObj, instanceId);
        state.reset();
    }

    void reset() override
    {
        if (m_handle)
        {
            m_handle->close();
            m_handle.reset();
        }
    }

private:
    bool isOpen() const
    {
        return m_handle && m_handle->isOpen();
    }

    void tryOpen(OgnMyIpcNodeCppDatabase& db)
    {
        // Open transport from db.inputs (e.g. URI, config).
        // m_handle = std::make_unique<MyTransportHandle>(std::string(db.inputs.uri()));
    }

    bool transfer(OgnMyIpcNodeCppDatabase& db)
    {
        // Non-blocking send or try-receive; write db.outputs on success.
        // See Performance Considerations for time-budget guidance.
        return false; // replace with actual transfer
    }

    bool ensureOpenAndTransfer(OgnMyIpcNodeCppDatabase& db)
    {
        if (!isOpen())
        {
            tryOpen(db);
            if (!isOpen())
                return false;
        }
        return transfer(db);
    }

    std::unique_ptr<MyTransportHandle> m_handle; // replace with your transport type
};

Python Node Implementation#

BaseResetNode는 isaacsim.core.nodes 익스텐션에서 제공됩니다. config/extension.toml에 의존성으로 추가하고 노드 파일에 임포트합니다:

[dependencies]
"isaacsim.core.nodes" = {}

import omni.graph.core as og
from isaacsim.core.nodes import BaseResetNode

소규모 클래스에 인스턴스별 데이터를 담아 BaseResetNode를 서브클래스화합니다. 샘플처럼 compute에서 소켓을 지연 열기하는 경우 super().__init__에 initialize=False를 전달합니다. 없으면 BaseResetNode.__init__이 생성 중 즉시 custom_reset()을 호출하는데, 이때 인스턴스 속성(예: self.sock = None)이 설정되기 전이므로 AttributeError가 발생합니다. 소켓을 닫고 버퍼를 지우도록 custom_reset()을 구현합니다. custom_reset()은 타임라인 정지 시 실행되며 C++의 reset()을 반영합니다.

생성된 OgnExamplePython 스텁을 다음과 같은 클래스로 교체합니다(완전한 TCP 구현은 isaacsim.examples.ipc의 OgnSimpleSendSimulationClockPy.py 참조):

import omni.graph.core as og
from isaacsim.core.nodes import BaseResetNode


class OgnMyIpcNodePyState(BaseResetNode):
    """Per-instance state for the template IPC node."""

    def __init__(self) -> None:
        # Declare all attributes BEFORE calling super().__init__,
        # because BaseResetNode.__init__ calls custom_reset() immediately.
        self.handle = None  # replace with your transport handle
        self.uri = ""
        super().__init__(initialize=False)

    def custom_reset(self) -> None:
        """Reset transport state when timeline or inputs change."""
        # Called on timeline stop and when inputs change.
        if self.handle is not None:
            self.handle.close()
            self.handle = None
        self.uri = ""


class OgnMyIpcNodePy:
    """Template OmniGraph node for custom IPC transports."""

    @staticmethod
    def internal_state() -> OgnMyIpcNodePyState:
        """Create per-instance state for the node."""
        return OgnMyIpcNodePyState()

    @staticmethod
    def compute(db: object) -> bool:
        """Evaluate one non-blocking IPC transfer step."""
        state = db.per_instance_state

        uri = db.inputs.uri
        if state.handle is not None and state.uri != uri:
            state.custom_reset()

        if state.handle is None:
            # Open transport from inputs (e.g. URI, config).
            # state.handle = open_my_transport(uri)
            # state.uri = uri
            # Fire execOut even on failure so downstream nodes keep running.
            db.outputs.execOut = og.ExecutionAttributeState.ENABLED
            return False

        # Non-blocking send or try-receive; write db.outputs on success.
        # See Performance Considerations for time-budget guidance.
        success = False  # replace with actual transfer

        # For send nodes: fire execOut every tick.
        # For receive nodes: fire execOut only when a full message arrives.
        if success:
            db.outputs.execOut = og.ExecutionAttributeState.ENABLED
        return success

     │    (recv: often "no full message yet")
     yes

     ▼

Building an Example Graph#

아래 단계는 isaacsim.examples.ipc의 참조 노드를 사용하여 tcp_tutorial_playback_bridge.py용 샘플 그래프를 구성합니다. 자신의 노드를 연결하기 전에 엔드-투-엔드 IPC 패턴을 확인하는 데 사용합니다.

  1. 샘플 익스텐션을 활성화합니다. Window > Extensions를 열고 isaacsim.examples.ipc를 검색하여 IPC OmniGraph Node Examples를 활성화합니다.

  2. Action Graph 에디터를 엽니다. Window > Graph Editors > Action Graph.

  3. 튜토리얼 노드를 배치합니다. 노드 라이브러리의 Isaac Examples 아래에 Receive External Step과 Send Simulation Clock을 추가합니다. 검색 박스를 사용하여 isaacsim.core.nodes에서 On Playback Tick과 Isaac Read Simulation Time을 추가합니다. C++ 또는 Python 노드 쌍 모두 브리지 스크립트와 작동합니다.

  4. 그래프를 배선합니다.

    실행 체인:

    • On Playback Tick execOut → Receive External Step execIn

    • Receive External Step execOut → Send Simulation Clock execIn

    데이터:

    • Isaac Read Simulation Time simulationTime → Send Simulation Clock simulationTime

    기본 uri 값은 수신 노드에서 127.0.0.1:9001, 발신 노드에서 127.0.0.1:9000입니다.

  5. 재생 시작. 툴바의 Play 버튼을 클릭하거나(또는 Space를 눌러) 시뮬레이션을 시작합니다.

일반 Action Graph UI는 Isaac Sim OmniGraph Tutorial과 Before You Start에 링크된 OmniGraph 문서에서 다룹니다.

그래프 배선 및 재생이 실행되면, Receive External Step이 URI에서 수신 대기하고, 브리지 스크립트가 연결하여 첫 번째 스텝 토큰을 전송하며, Send Simulation Clock이 매 틱 후 스크립트에 현재 시뮬레이션 시간을 보고합니다. 스크립트가 타이밍 루프를 구동하고 Isaac Sim은 수신된 스텝당 한 틱씩 진행합니다.

시도하기: 자신의 노드로 브리지 실행

isaacsim.examples.ipc 노드로 참조 그래프가 엔드-투-엔드로 작동하면 커스텀 노드로 교체합니다:

  1. 그래프에서 SimpleSendSimulationClock 노드를 삭제합니다.

  2. 위 연습에서 이름을 바꾼 노드를 추가합니다.

  3. 동일한 방식으로 배선합니다: Receive External Step execOut → 자신의 노드 execIn, Isaac Read Simulation Time simulationTime → 자신의 노드 simulationTime.

  4. 브리지 스크립트를 실행합니다. transfer()가 여전히 데이터를 전송하지 않고 true를 반환하는 스텁이므로, 스크립트는 연결되지만 클록 출력을 받지 못합니다 — 이는 예상된 동작입니다. 이것으로 익스텐션이 로드되고 활성화되며 그래프에 참여함을 확인합니다.

  5. 구현을 완성하려면 transfer()에 실제 전송 로직을 추가합니다. source/extensions/isaacsim.examples.ipc/의 OgnSimpleSendSimulationClockCpp.cpp(또는 Python 버전)를 참조로 사용합니다.

External Python Playback Bridge#

tcp_tutorial_playback_bridge.py 스크립트는 완전한 왕복을 보여줍니다. Send 노드가 방출하는 8바이트 클록을 수신하고, Receive 노드의 수신 대기 포트에 연결하고, 하나의 스텝을 준비한 다음 각 프레임에서 클록을 읽고 다음 OnPlaybackTick이 발동될 수 있도록 다음 스텝을 전송합니다.

스크립트는 Python 표준 라이브러리(socket, struct, argparse)만 사용하며 Isaac Sim이나 서드파티 의존성이 없습니다. 시스템 python3으로 리포지토리 루트에서 실행합니다:

python3 source/extensions/isaacsim.examples.ipc/python/scripts/tcp_tutorial_playback_bridge.py

--help를 전달하면 --clock-host, --clock-port, --step-host, --step-port, --max-frames 옵션을 볼 수 있습니다.

Warning

스크립트는 127.0.0.1에 TCP 리스너를 바인딩합니다. 실제 배포에서는 의도적으로 포트를 노출하지 않는 한 루프백에만 바인딩합니다. 열린 인터페이스는 공격 표면을 증가시킬 수 있습니다. IPC 브리지는 네트워크 서비스처럼 취급하여 인증, TLS 또는 동등한 수단, 방화벽 규칙은 사용자의 책임입니다.

Performance Considerations#

프레임 예산 내에 유지합니다.

OmniGraph는 시뮬레이션, UI 및 다른 그래프에 대해 응답성을 유지해야 하는 경로에서 compute를 평가합니다. 일반적인 실패 모드는 예측할 수 없이 긴 작업이며 그 자체로 "동기식" I/O가 아닙니다. 느린 상대방 대기, 대용량 복사, 경합 잠금, 또는 수십 밀리초 동안 중단될 수 있는 RPC는 성능 문제를 유발할 수 있습니다.

작은 빠른 경로는 종종 괜찮습니다.

compute의 작고 고정 크기의 fire-and-forget 작업(소켓 연결 후 튜토리얼의 8바이트 클록 전송)은 목표 프레임 레이트에서 노드당 예산 내에 일관되게 완료되면 그래프 스레드에 유지할 수 있습니다. 경로를 측정했고 원격 측의 백프레셔를 기다리지 않는 경우 다른 스택에도 동일하게 적용됩니다.

워커, 큐, 또는 비동기 API를 사용할 때.

  • 호출이 알 수 없는 시간 동안 차단될 수 있는 경우(요청/응답, 준비 대기, 대용량 페이로드, 또는 노드당 예산을 초과할 수 있는 모든 것), 해당 IPC를 워커 스레드에서 실행합니다. 결과를 큐에 넣는 콜백을 사용하고 compute는 비차단 디큐와 db.outputs 쓰기만 담당합니다.

  • 인바운드 데이터의 경우 수신 시도(튜토리얼의 스텝 노드처럼)는 외부 프로세스가 사용자 일정에 따라 전송하지 않을 때 무기한 대기를 피합니다.

  • 비동기 또는 콜백 기반 I/O: 워커 스레드에서 네트워크 또는 IPC를 구동하고, 디코딩된 메시지를 스레드 안전 큐에 넣고, compute는 비차단 디큐와 db.outputs 쓰기만 담당합니다.

  • 지연된 완료: compute에서 응답을 기다리지 않고 작업을 게시합니다. 백그라운드 스레드가 결과를 나중 평가를 위해 큐에 넣습니다.

구조화된 메시지 대 고정 바이트.

튜토리얼의 고정 크기 프레이밍은 명확성을 위한 것입니다. 프로덕션 브리지는 일반적으로 라이브러리의 메시지 형식(IDL 생성 타입, JSON, 또는 다른 스키마)을 사용합니다. 언제 전송할지, 인바운드 데이터를 어떻게 파싱할지, 각 compute를 예산 내로 유지하는 방법은 여전히 사용자가 결정합니다.

대용량 메시지(카메라 프레임, 포인트 클라우드).

멀티 메가바이트 페이로드를 이동하는 단발성 호출은 메모리와 스케줄링에 부담을 줄 수 있습니다. 스트리밍 API, 명시적 백프레셔(느린 소비자에서 프레임 드롭 또는 건너뜀), 또는 노드가 핸들이나 소규모 메타데이터만 전달하는 OmniGraph 외부의 공유 메모리 및 제로 카피 경로를 사용합니다.

Built-In Nodes for Data in and Out#

isaacsim.examples.ipc 외에도, 여러 익스텐션이 일반 목적 브리지 역할 없이 Isaac Sim 내에서 시뮬레이션 상태를 읽거나 시뮬레이션을 구동하는 OmniGraph 노드를 등록합니다. 표는 브리지 그래프에서 커스텀 IPC 노드 옆에 자주 위치하는 타입을 강조합니다.

커스텀 노드의 입력과 출력을 설계하기 전에 연결하려는 빌트인 노드의 .ogn을 확인하세요 — 출력 속성 이름과 타입이 노드가 소비하거나 생성해야 하는 것을 결정합니다.

브리지 스타일 그래프에 자주 사용되는 빌트인 OmniGraph 노드#

목적

노드(등록된 타입)

익스텐션

주요 입력/출력(약식)

isaacsim.core.nodes

입력: robotPath 또는 targetPrim, 선택적 jointNames/jointIndices. 출력: jointPositions, jointVelocities(double[]), jointNames(token[]), 측정된 노력 배열 포함.

대안 관절 상태(물리 센서 경로)

isaacsim.sensors.physics.IsaacReadJointState

isaacsim.sensors.physics.nodes

입력: prim(아티큘레이션 루트). 출력: jointPositions, jointVelocities, jointEfforts, jointNames, execOut 등.

입력: 위와 동일한 로봇 타겟팅; positionCommand, velocityCommand, effortCommand(double[]). 각도 단위는 컨트롤러 API에서 라디안.

OnPhysicsStep, IsaacSimulationGate, IsaacReadSimulationTime, …

상태 읽기와 명령 쓰기를 일관되게 속도 조절하는 데 사용합니다(정확한 속성은 노드마다 다름).

카메라/뷰포트 렌더 제품 경로(설정만)

주로 경로와 타겟(renderProductPath, renderProductPrim). 픽셀은 별도의 읽기 단계가 필요합니다. Camera and Render Products 참조.

커스텀 발신기 앞에 연결할 수 있는 다른 읽기 익스텐션:

  • isaacsim.sensors.physics.nodes — isaacsim.sensors.experimental.physics가 지원하는 IMU, 접촉, 노력 등.

  • isaacsim.sensors.physx — 예: Isaac Read Lidar Beams, Isaac Read Lidar Point Cloud, Isaac Read Light Beam Sensor.

  • isaacsim.sensors.rtx.nodes — 예: Isaac Extract RTX Sensor Point Cloud.

외부 애플리케이션(토픽, 서비스, 또는 다른 런타임)과의 IPC에는 위 표의 노드를 전송 계층으로 취급하는 대신 전용 브리지 익스텐션을 사용합니다. 예: isaacsim.ros2.nodes(ROS 2) 또는 isaacsim.ucx.nodes(UCX). 이러한 익스텐션은 표의 센서 읽기 노드가 아닌 TCP 튜토리얼 노드와 동일한 역할을 합니다.

이 저장소의 참조 구현.

source/extensions/ 디렉토리에는 TCP 튜토리얼을 넘어섰을 때 참고할 수 있는 완전한 IPC 브리지 구현이 포함되어 있습니다. 두 가지 스택이 있습니다:

  • ROS 2: isaacsim.ros2.nodes, isaacsim.ros2.bridge 및 관련 패키지.

  • UCX: isaacsim.ucx.nodes, isaacsim.ucx.core, isaacsim.ucx.bridge.

각 스택은 완전한 브리지 레이아웃을 보여줍니다: extension.toml 의존성, 네이티브 플러그인, C++ 및 Python OmniGraph 노드, 전송 백엔드.

Kit 문서의 OmniGraph 노드 라이브러리에서 이름으로 검색합니다: OmniGraph.

Camera and Render Products#

커스텀 IPC 노드에 원시 RGB 픽셀을 가져오려면 단순한 uchar[] OGN 입력 이상이 필요합니다. 이미지는 일반적으로 IPC 인코더에 도달하기 전에 Replicator 파이프라인 또는 렌더 제품 체인을 통해 흐르며 그래프 에디터의 단일 와이어가 아닙니다. 주요 단계는 다음과 같습니다:

  1. 렌더 제품(IsaacCreateRenderProduct 또는 IsaacGetViewportRenderProduct)을 설정하고 카메라를 연결합니다.

  2. 렌더 제품을 읽기 메커니즘에 공급합니다:

    • Replicator 어노테이터(호스트 친화적인 NumPy 배열)

    • Hydra 텍스처 체인(IsaacAttachHydraTexture를 통한 GPU 핸들)

  3. 결과 CPU 주소 지정 가능 바이트 또는 배열을 IPC 인코더 노드에 전달합니다.

isaacsim.ros2.bridge 익스텐션의 카메라 헬퍼 노드는 이 파이프라인이 어떻게 조립되는지에 대한 구체적인 참조입니다. ROS 2 카메라 퍼블리셔는 렌더 제품을 호스트 읽기에 연결한 다음 IPC에 연결합니다. 해당 소스를 탐색하는 것이 자신의 것을 구성하기 전에 패턴을 이해하는 가장 빠른 방법입니다.

compute를 통해 대용량 버퍼를 전달하기 전에 Performance Considerations를 참조합니다. 카메라 프레임은 프레임 예산 초과의 일반적인 원인입니다.

Testing Your OmniGraph Node Implementation#

OmniGraph 노드에 대한 Python 통합 테스트는 런타임에 og.Controller.edit를 사용하여 Action Graph를 구성하고, 노드를 프로그래밍 방식으로 연결하고, 타임라인을 구동하며, 출력 속성 값을 검증할 수 있습니다. 커버해야 할 유용한 영역:

  • 정확한 출력: 알려진 입력이 주어지면 노드가 예상된 db.outputs 값을 생성합니다.

  • execOut 타이밍: 노드가 의도된 조건에서만 execOut을 발동합니다. 발신 노드의 경우 매 평가, 수신 노드의 경우 데이터 수신 시에만.

  • 리셋 동작: URI 입력 변경 또는 타임라인 정지 시 전송이 닫히고, 이후 평가에서 깨끗하게 다시 열립니다.

  • 엣지 케이스: 부분 메시지, 상대방 연결 해제, 외부 프로세스의 잘못된 형식 데이터.

C++ 헬퍼(파싱, 인코딩, 엔디안)의 경우 단위 테스트는 doctest와 같은 네이티브 테스트 라이브러리를 통해 Isaac Sim 외부에서 실행할 수 있으며, premake5.lua에 연결하고 extension.toml에서 참조합니다.

extension.toml의 [[test]] 항목이 테스트 모듈을 가리키도록 합니다. 생성된 테스트 드라이버는 일반적으로 _build/<platform>/<config>/tests/tests-<your.extension.id>.sh입니다. 스캐폴딩된 익스텐션의 경우 테스트는 python/tests/에 있습니다(템플릿에서 이미 생성됨).

위 패턴의 예제(비동기 테스트, OnImpulseEvent, 자유 포트 헬퍼, 타임라인 제어)는 source/extensions/isaacsim.examples.ipc/python/tests/를 참조합니다.
