Isaac Sim Performance Optimization Handbook#

Understanding Isaac Sim Performance#

시뮬레이션 속도는 다양한 요인에 의해 영향을 받을 수 있습니다. 물리 스텝 크기, 장면의 복잡도, 충돌체의 수와 복잡도, GPU 동역학 사용 여부 등이 요인에 포함됩니다.

Note

KPI 측정 페이지에서 주요 성능 지표를 측정하는 방법을 확인하세요.

Note

성능 병목 지점을 파악하려면 시뮬레이션 프로파일링이 도움이 됩니다. Profiling을 참조하세요.

Physics Simulation Optimizations#

  1. Physics Step Size: 물리 스텝 크기는 각 물리 계산 단계의 시간 간격을 결정합니다.

  • 스텝 크기가 작을수록 시뮬레이션이 더 정확해지지만 더 많은 계산 리소스가 필요합니다.

  • 스텝 크기가 크면 시뮬레이션이 빨라지지만 물리 정확도가 떨어질 수 있습니다.

Note

스크립트에서 set_physics_dt()를 사용하여 물리 스텝 크기를 조정하세요.

  1. PhysX Minimum Frame Rate Clamp (/physics/minFrameRate): PhysX가 누적된 물리 시간의 최대량을 제한합니다. 렌더링 프레임이 물리 스텝보다 훨씬 느리게 실행될 때 무한한 물리 업데이트를 방지합니다.

  • 클램프 올리기(예: 60으로)는 물리 단계가 더 자주 실행될 수 있도록 허용합니다. 렌더링 FPS가 물리 Hz보다 느릴 때 유용합니다.

  • 클램프 낮추기(예: 10으로)는 각 렌더 프레임당 물리 단계 수를 제한하여 렌더링이 물리보다 느릴 때 시뮬레이션이 뒤처지는 것을 방지합니다.

이 설정은 타임라인의 dt와 동일하지 않습니다. 이는 렌더링 프레임 대 물리 업데이트 관계에 대한 안전 장치입니다.

Note

/physics/minFrameRate carb 설정을 수정하여 PhysX 최소 프레임 레이트 클램프를 조정하세요.

  1. GPU Dynamics: GPU 동역학을 활성화하면 물리 계산을 CPU에서 GPU로 오프로드하여 시뮬레이션 속도를 높일 수 있습니다.

Note

이는 GPU가 충분히 강력하고 다른 작업으로 이미 완전히 사용되지 않는 경우에만 유익합니다.

  1. Physics Scene Complexity: 장면의 물리 객체 복잡도는 시뮬레이션 성능에 영향을 줍니다.

  • 간단한 충돌체가 일반적으로 가장 성능이 좋습니다. 성능 순서는 다음과 같습니다:

    • 기본 충돌체(박스, 구, 캡슐, 평면)가 가장 성능이 좋습니다.

    • 볼록 메시가 다음으로 성능이 좋습니다(Convex Hull 또는 Convex Decomposition 근사)

    • 실린더는 부드럽고 정확한 구름 동작에 좋은 선택이지만 저폴리곤 볼록 근사보다 비쌉니다.

  • 시뮬레이션하는 워크플로에 필수적이지 않은 충돌체를 비활성화하거나 단순화하세요. 예를 들어, 조작 태스크에서 로봇 팔꿈치 메시에 충돌 기하학이 필요한지 고려하세요.

    • 마찬가지로 불필요한 충돌을 피하세요. 가능하면 객체 겹침 수를 줄여 충돌 감지 및 해결 계산을 최소화하세요.

Note

이는 장면 전체와 개별 물리 객체 모두에 적용됩니다. 동적 물체(비정적 객체)의 복잡한 충돌체는 더 큰 영향을 미칩니다.

  1. Adjusting PhysX Thread Count: PhysX에서 사용하는 스레드 수는 최적 성능을 위해 조정될 수 있습니다.

Note

이는 특히 CPU 기반 물리 시뮬레이션에 적용됩니다. 스레드 수를 0으로 낮추면 메인 스레드에서 실행됩니다.

PhysX 성능 문서를 확인하세요.

Robot Asset Optimizations#

OpenUSD Tuning Tutorials에서 샘플 에셋 최적화 단계별 튜토리얼을 제공합니다.

  1. Merge Mesh Tool: Tools > Robotics의 Merge Mesh 도구를 사용하면 물리 계산이나 로보틱스 애플리케이션에 관련 없는 시각적 세부 사항을 단순화할 수 있습니다.

  2. Scenegraph Instancing: 인스턴싱은 공유 가능하고 참조 가능한 prim 서브그래프를 가능하게 합니다. 동일한 객체 또는 유사한 객체가 많은 장면에서 메모리를 절약할 수 있습니다.

    • 인스턴싱의 예시는 여기에 설명되어 있습니다.

Note

인스턴싱은 본질적으로 속성 관련 제한을 수반합니다. 자식 노드는 수정된 속성을 가질 수 없으며 자신만의 속성을 추가할 수 없습니다.

  1. Simplify Colliders: 충돌체는 높은 계산 비용이 발생합니다. 충돌체가 단순할수록 성능이 좋습니다.

    • 접촉 지점을 줄이면 성능이 크게 향상됩니다. 휠 충돌체의 경우 구 충돌체가 캡슐보다 접촉 지점이 적어 더 빠릅니다.

    • 로봇의 경우 필요한 정밀도를 제공하는 가장 단순한 근사를 사용하세요. 예를 들어, 팔 링크에는 기본 캡슐 또는 박스 충돌체를 사용하세요.

    • 충돌체의 총 수를 줄이는 것도 유익합니다. 워크플로에 추가된 모든 충돌체가 실제로 필요한지 고려하세요.

Note

높은 정밀도 애플리케이션에는 단순화된 형태보다 메시 충돌체를 사용해야 합니다. 트레이드오프가 있습니다.

  1. Disable Self Collisions: Articulation Root에서 자기 충돌을 비활성화하면 관절형 물체의 시뮬레이션 속도를 높일 수 있습니다.

Note

이는 사용 사례에 크게 의존합니다. 복잡한 관절형 손에서는 자기 충돌이 현실적인 파지를 위해 필요합니다.

Scene and Rendering Optimizations#

사용 가능한 렌더러 모드와 각각을 사용해야 하는 경우에 대한 개요는 Rendering Modes를 참조하세요.

  1. Simplify the Scene: 장면의 복잡도를 줄이고, 레벨 오브 디테일(LOD) 기법을 구현하고, 씬 최적화 도구를 활용하면 성능을 향상시킬 수 있습니다.

Note

Isaac Sim은 장면 단순화를 위한 여러 도구를 제공합니다.

  • Scene Optimizer

  • Merge Mesh

Note

실시간 시뮬레이션 중 기즈모(바닥 그리드 포함)는 성능 최적화를 위해 자동으로 사라집니다.

  1. Using RTX Real-Time 2.0: (RTX Real-Time 2.0 참조)

    • RT2는 Isaac Sim의 새로운 기본 렌더링 모드입니다. 기존 RTX Real-Time에 비해 정확도와 성능이 모두 향상되었습니다.

    • 루프 내 훈련 또는 처리량을 완전한 광선 전송보다 우선시하는 기타 워크플로의 경우, 경로 추적이나 RT2 대신 레스터화 모드를 고려하세요.

    • 레트레이스 임계값을 낮추면 약간 더 편향된 결과를 감수하고 성능을 향상시킬 수 있습니다.

      "--/rtx/pathtracing/cached/retrace=0.1"

    • Fractional Cutout Opacity를 비활성화하면 반투명 재료의 정확도를 잃는 대신 성능을 향상시킬 수 있습니다.

      "--/rtx/pathtracing/fractionalCutoutOpacity=false"

Note

일부 구형 하드웨어, 특히 Ampere 세대에서는 RT2 성능이 RTX Real-Time보다 낮을 수 있습니다. 이 경우 레스터화 모드를 사용하는 것이 좋습니다.

  1. Disable Materials and Lights: (Performance Modes 참조)

    • 장면에 있는 많은 수의 재료로 인해 로드 시간이 크게 느려질 수 있습니다. 렌더링 모드를 낮추면 재료를 비활성화하여 로드 시간을 단축할 수 있습니다.

      "--/app/renderer/skipMaterialLoading=true"

    • 조명을 비활성화하면 렌더링 부하를 단순화하고 성능을 향상시킬 수 있습니다.

    • 렌더 설정 패널에서 렌더링 기능을 끄세요(이것들은 동등한 carb 설정도 있습니다).

  2. Adjust DLSS Performance Mode: DLSS 성능 모드는 /persistent/rendering/dlss/execMode carb 설정으로 전환됩니다.

    • Performance (0) - 가장 성능 중심의 모드. 더 많은 에일리어싱이 발생할 수 있습니다.

    • Balanced (1) - 성능과 품질의 균형을 제공합니다.

    • Quality (2) - 고품질 모드를 제공하지만 성능이 낮습니다.

    • Auto (3) - 해상도에 따라 최적 모드를 선택합니다.

Note

현재 DLSS 모드는 기본적으로 Performance로 설정되어 있습니다.

  1. Disabling viewport updates in headless mode: SimulationApp이 있는 헤드리스 모드에서 실행할 때 뷰포트 업데이트가 발생할 수 있습니다.

    • 성능을 향상시키려면 omni.kit.viewport.utility.get_active_viewport().updates_enabled = False를 설정하여 뷰포트 업데이트를 비활성화하세요.

      from isaacsim import SimulationApp

simulation_app = SimulationApp({"headless": True, "disable_viewport_updates": True})

    • SimulationApp을 사용하지 않는 경우 다음 코드 스니펫으로 뷰포트 업데이트를 비활성화할 수 있습니다.

      from omni.kit.viewport.utility import get_active_viewport

viewport = get_active_viewport()
viewport.updates_enabled = False

Note

disable_viewport_updates 플래그를 사용하면 카메라 API 수동 렌더링이 여전히 가능합니다.

  1. Disabling texture streaming: 텍스처 스트리밍은 GPU 메모리 사용량을 최소화하는 기능으로, 렌더링되는 텍스처만 GPU에 로드됩니다.

    • 텍스처 스트리밍을 비활성화하면 성능상 이점이 있을 수 있지만 GPU 메모리 사용량이 증가합니다.

    • /rtx/resourcemanager/enableTextureStreaming carb 설정 값을 수정하여 텍스처 스트리밍을 비활성화하세요.

      "--/rtx-transient/resourcemanager/texturestreaming/enabled=false"

Note

모든 사용 사례에 권장되지 않습니다. 사례별로 평가 후 사용해야 합니다.

CPU Thread Count Optimizations#

Isaac Sim이 사용하는 CPU 스레드 수를 제어하는 세 가지 설정이 있습니다. 설정하지 않으면(기본 동작) Isaac Sim이 자동으로 결정합니다.

  1. --/plugins/carb.tasking.plugin/threadCount

  2. --/persistent/physics/numThreads

  3. --/plugins/omni.tbb.globalcontrol/maxTBBThreads

너무 많은 워커 스레드를 생성하면 CPU 병목 현상이 발생할 수 있습니다. 시스템에서 사용 가능한 CPU 코어 수로 CPU 스레드 수를 제한하는 것을 고려하세요.

예를 들어, Ubuntu에서:

./isaac-sim.sh --/plugins/carb.tasking.plugin/threadCount=16 --/plugins/omni.tbb.globalcontrol/maxThreadCount=16

Standalone Python:

from isaacsim import SimulationApp

simulation_app = SimulationApp({"headless": False, "limit_cpu_threads": 16})

CPU Governor Settings on Linux#

CPU 거버너는 CPU의 작동 클록 주파수 범위 및 스케일링을 결정합니다. 이는 성능에 제한이 될 수 있습니다.

sudo apt-get install linux-tools-common
cpupower frequency-info # Check available governors
sudo cpupower frequency-set -g performance # Set governor with root permissions

Note

모든 시스템에서 모든 거버너를 사용할 수 있는 것은 아닙니다. 더 높은 클록 속도를 가능하게 하는 거버너는 일반적으로 특별한 시스템 권한이 필요합니다.

Asynchronous Rendering#

비동기 렌더링은 렌더링이 시뮬레이션 업데이트와 별도의 스레드에서 실행되도록 하는 기능으로, 더 나은 CPU/GPU 사용을 허용합니다.

Asynchronous Rendering Toggle (Default)#

이는 isaacsim.core.throttling 익스텐션에서 설정됩니다. 예상치 못한 동작이 발생할 경우 이 기능을 비활성화하세요.

./isaac-sim.sh --exts."isaacsim.core.throttling".enable_async=false

Note

이 설정은 SimulationApp을 사용하여 실행할 때만 true로 설정됩니다.

Runtime Asynchronous Rendering (Experimental)#

비동기 렌더링은 런타임 중에 실험적으로 지원됩니다. 런타임에 비동기 렌더링을 활성화하려면 다음 설정을 사용하세요.

./isaac-sim.sh --exts."isaacsim.core.throttling".enable_async=false --/app/asyncRendering=true --/app/omni.usd/asyncHandshake=true --/omni/replicator/asyncRendering=true

./python.sh script.py --/app/asyncRendering=true --/app/omni.usd/asyncHandshake=true --/omni/replicator/asyncRendering=true

Note

이 기능은 실험적이며 예상치 못한 동작을 초래할 수 있습니다. 이 기능을 활성화한다고 해서 반드시 FPS가 향상되는 것은 아닙니다.

Multi-GPU Support#

다음 경험 법칙이 멀티 GPU 성능을 향상시키는 데 도움이 될 수 있습니다. 멀티 GPU 벤치마크를 기반으로 합니다.

Note

여러 데이터 센터급 GPU를 사용할 때의 정확한 Isaac Sim 성능 지표는 여기에서 찾을 수 있습니다.

  1. 렌더링되는 카메라 수만큼 GPU를 추가하되 그 이상은 추가하지 마세요. 2개의 720p 카메라를 렌더링할 때 GPU를 2개에서 4개로 늘리면 성능이 향상되지 않습니다.

  2. 카메라가 더 많을수록 성능이 더 잘 스케일됩니다. 4개의 카메라와 4개의 GPU를 사용하면 카메라당 GPU를 사용하는 것보다 2개의 GPU로 2개의 카메라를 렌더링하는 것보다 더 큰 성능 향상을 보입니다.

  3. 단일 고해상도 카메라는 여러 GPU에서 더 빠르게 렌더링됩니다. 경험 법칙의 예외로, 단일 4K 카메라는 여러 GPU에 걸쳐 스케일됩니다.

  4. GPU 수를 늘려도 장면 로드 시간이 개선되지 않습니다. GPU 수는 로드 시간에 영향을 미치지 않습니다.

  5. GPU 물리 시뮬레이션은 1개의 GPU만 활용합니다. GPU 수를 늘려도 GPU 기반 물리 시뮬레이션이 향상되지 않습니다.

Disabling IOMMU On Linux#

CUDA 프로그래밍 가이드에 따르면, IOMMU가 활성화된 경우 Linux의 P2P 전송이 지원되지 않습니다.

IOMMU는 시스템 BIOS에서 비활성화할 수 있습니다(정확한 지침은 마더보드 사양에 따라 다릅니다).

sudo bash -c 'echo GRUB_CMDLINE_LINUX="amd_iommu=off" >> /etc/default/grub'
sudo update-grub
sudo reboot

재부팅 후 IOMMU가 비활성화되어야 합니다.

Note

IOMMU가 활성화된 경우 다음과 같은 경고가 표시됩니다: NVRM: GPU at PCI:0000:XX:00: (...

Reducing GPU Memory Utilization#

다음은 Isaac Sim GPU 메모리 사용량을 줄이는 데 도움이 되는 제안입니다:

  1. 텍스처 스트리밍 예산을 줄이세요. Isaac Sim 설정에서 Texture Budget (MB)를 줄이세요.

  2. 불필요한 뷰포트를 끄거나 렌더링된 카메라 해상도를 줄여 총 렌더링 픽셀 수를 줄이세요.

Experimental: Reducing Potential Memory Leaks#

긴 실행 워크플로에서 메모리 누수가 발생하는 경우 Isaac Sim RAM 소비를 줄이기 위한 실험적 제안:

  1. 특히 단계가 종료될 때 Isaac Sim이 로드된 상태에서 스테이지가 열리고 닫히는 워크플로에서 메모리 누수를 줄이기 위해 메모리 할당자를 조정하세요.

    • 할당자 구성을 변경하려면 다음 환경 변수를 설정하세요:

      export GLIBC_TUNABLES=glibc.malloc.arena_max=1:glibc.malloc.mmap_max=0:glibc.malloc.mmap_threshold=2147483647

Useful Tools#

Windows#

Task Manager는 다양한 변수의 최고 사용량을 표시할 수 있는 깔끔한 그래프를 제공하는 유용한 리소스입니다.

  1. 시작 아이콘을 클릭합니다.

  2. "Task Manager"를 입력합니다.

  3. Task Manager에서 "Performance" 탭을 선택합니다.

이 창의 왼쪽에는 CPU, 메모리, GPU 등의 다양한 그래프가 표시됩니다. 원하는 항목을 선택하면 세부 정보를 확인할 수 있습니다.

Linux#

nvidia-smi는 Linux에서 GPU 메모리 사용량을 모니터링하기 위한 좋은 리소스입니다.
