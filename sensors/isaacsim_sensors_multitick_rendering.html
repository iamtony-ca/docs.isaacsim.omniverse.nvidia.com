Multi-Tick Rendering#

멀티틱 렌더링은 각 센서의 렌더 속도를 메인 시뮬레이션 프레임 속도로부터 분리합니다. 매 프레임마다 모든 센서를 렌더링하는 대신, 각 센서는 독립적으로 자체 설정 속도로 틱을 발생시킵니다. 이를 통해 불필요한 렌더링 작업을 방지하여 많은 센서가 있는 장면에서 성능이 크게 향상됩니다.

멀티틱 렌더링은 Isaac Sim 6.0 이상에서 기본적으로 활성화되어 있습니다.

Overview#

Isaac Sim 6.0 이전에는 모든 카메라와 RTX 센서가 시뮬레이션 프레임 속도로 렌더링되었습니다. 멀티틱 렌더링이 활성화되면, 렌더러는 각 센서에 대해 독립적인 틱 카운터를 유지하고 센서의 틱 인터벌이 경과했을 때만 렌더링합니다.

각 센서 프림의 omni:sensor:tickRate 속성은 렌더 빈도를 Hz 단위로 제어합니다. 값이 0(기본값)이면 센서를 자동트리거 모드로 설정하여, 멀티틱 렌더링이 비활성화된 것처럼 매 프레임마다 렌더링합니다.

Performance Benefits#

  • GPU 부하 감소: 매 프레임마다 업데이트할 필요가 없는 센서는 렌더링을 완전히 건너뛰어 다른 센서나 시뮬레이션 작업을 위한 GPU 리소스를 확보합니다.

  • 독립적인 속도: 60Hz로 동작하는 입체 깊이 카메라가 더 이상 30Hz RGB 카메라도 같은 속도로 렌더링하도록 강제하지 않습니다.

  • 향상된 확장성: 많은 센서를 사용하는 장면(예: 멀티 로봇 플릿)에서 각 센서가 실제로 틱을 발생시킬 때만 GPU 시간을 소비하므로 비례적으로 더 큰 개선 효과를 얻습니다.

Configuring Per-Sensor Tick Rates#

센서 틱 속도는 Camera와 OmniLidar 프림에 적용되는 OmniSensorAPI USD 스키마의 omni::sensor::tickRate 속성으로 제어됩니다. Isaac Sim 6.0의 기본 제공 USD 애셋에는 이미 이 스키마가 적용되어 적절한 기본 틱 속도가 설정되어 있습니다.

Note

OmniSensorAPI 스키마는 OmniRadar 프림에도 적용되지만, Isaac Sim 6.0 GA에서 RTX Radar 렌더러는 omni:sensor:tickRate를 무시하고 항상 자동트리거합니다. RTX Radar autotriggers in 6.0 GA를 참조하십시오.

Using the Isaac Sim Extension API#

isaacsim.sensors.experimental.rtx 확장은 센서 프림을 작성하고 틱 속도를 설정하기 위한 Python API를 제공합니다. 예를 들어, /World/Lidar에 10Hz 틱 속도로 OmniLidar 프림을 작성하려면:

from isaacsim.sensors.experimental.rtx import Lidar

# Render the Lidar at 10 Hz independently of the simulation frame rate.
lidar = Lidar(path="/World/Lidar", tick_rate=10.0)

카메라의 경우:

from isaacsim.sensors.experimental.rtx import RtxCamera

# Render the Camera at 30 Hz independently of the simulation frame rate.
camera = RtxCamera(path="/World/Camera", tick_rate=30.0)

Using OmniGraph#

ROS2, UCX, 또는 HSB 헬퍼 OmniGraph 노드를 사용할 때, 센서 틱 속도는 센서 프림의 omni:sensor:tickRate 속성에서 읽어옵니다. 추가적인 노드 구성은 필요하지 않습니다.

시뮬레이션 프레임 속도에 맞춰 센서 및 그래프 발행 속도를 조정하는 ROS2 예제는 ROS2 Setting Publish Rates를 참조하십시오.

OmniLidar Tick Rate Must Equal scanRateBaseHz#

Warning

OmniLidar 프림의 경우, 스캔 누적과 멀티틱 렌더링이 올바르게 동작하려면 omni:sensor:tickRate는 반드시 omni:sensor:Core:scanRateBaseHz와 동일하게 설정해야 합니다.

두 값이 다르면, Lidar 모델은 전체 회전(회전식 Lidar) 또는 전체 방위각 스윕(고체 상태 Lidar)을 누적하는 대신 매 프레임마다 부분 스캔을 생성하는 폴백으로 전환됩니다. 프레임당 전체 스캔을 가정하는 다운스트림 파이프라인들—ROS2 RTX Lidar Helper laser_scan 발행자, IsaacComputeRTXLidarFlatScan, 및 누적된 GenericModelOutput 데이터의 모든 소비자—는 자동으로 잘린 출력을 보게 됩니다. 렌더러는 이 경우 오류를 기록하지 않습니다.

예를 들어, 기본 제공 Example_Rotary Lidar 구성은 scanRateBaseHz = 10이므로, 센서는 10Hz로 틱을 발생시켜야 합니다:

from isaacsim.sensors.experimental.rtx import Lidar

# Render at 10 Hz regardless of simulation frame rate.
lidar = Lidar.create("/World/Lidar", config="Example_Rotary", tick_rate=10.0)

같은 제약이 Example_Solid_State와 RTX Lidars의 벤더 구성에도 적용됩니다. 기존 OmniLidar 프림을 래핑할 때는 프림에서 omni:sensor:Core:scanRateBaseHz를 읽어 해당 값을 tick_rate로 전달하십시오.

Architecture: Timeline, Physics, and the Renderer#

멀티틱 렌더링이 활성화되면, 세 개의 클럭이 omni.kit.app 업데이트당 독립적으로 진행됩니다. 이 섹션에서는 각 클럭이 무엇인지, 어떻게 진행되는지, 그리고 서로 어떻게 상호작용하는지 설명합니다. omni.timeline 시간을 물리 또는 센서 시간과 혼합하는 코드는 일부 구성에서 일관되지 않은 타임스탬프를 생성할 수 있으므로, 결정론적 시뮬레이션을 구성하거나 센서 출력을 해석할 때 이러한 관계가 중요합니다.

The three clocks#

클럭

진행 소스

제어 주체

읽기 주체

실행 루프 / omni.timeline 시간

앱 업데이트당 벽시계 dt, 또는 수동으로 설정된 경우 고정 수동 dt

RenderingManager.set_dt(...) (수동 모드), /app/runLoops/main/rateLimitFrequency

Timeline UI, timeCodesPerSecond, 스테이지 업데이트 스케줄링

물리 시뮬레이션 시간

stepCount / stepsPerSecond로 매 물리 스텝마다 재계산됨

PhysicsScene.set_dt(...) 또는 SimulationManager.setup_simulation(dt=...)

SimulationManager.get_simulation_time(), IsaacReadSimulationTime, ROS 2 / UCX / HSB 타임스탬프

렌더러 시뮬레이션 시간

물리 시간의 미러로, 매 물리 스텝 후 Fabric 프림 /ExternalSimulationTime.omni:time에 기록됨

isaacsim.core.simulation_manager에 의해 구동됨; 시뮬레이션 매니저가 없을 때는 RenderingManager에 의해 시딩됨

eHydraRendering의 멀티틱 렌더러 센서당 틱 스케줄러

렌더러는 더 이상 현재 시뮬레이션 시간을 얻기 위해 omni.timeline을 읽지 않습니다; 대신 Fabric의 /ExternalSimulationTime 프림의 omni:time 속성을 읽습니다. 물리는 스테이지 업데이트 순서 -10에서 모든 onPhysicsStep 콜백에서 해당 속성을 기록하므로, 최신 값은 같은 앱 업데이트 내 순서 30에서 Hydra에서 사용할 수 있습니다.

Per-frame ordering#

타임라인이 재생 중인 하나의 App.update()의 경우:

  1. loop_dt(수동 또는 벽시계)를 계산하고 omni.timeline은 loop_dt만큼 진행합니다.

  2. 물리 스테이지 업데이트 단계는 루프 시간을 따라잡기 위해 physics_dt의 N >= 0 서브스텝을 실행합니다. 각 서브스텝은 물리 시뮬레이션 시간을 재계산하고 /ExternalSimulationTime에 기록합니다.

  3. Hydra는 /ExternalSimulationTime을 한 번 읽습니다. 센서당 틱 스케줄러는 해당 값을 각 센서의 마지막 렌더링 시간 및 omni:sensor:tickRate와 비교하여 이 프레임에서 렌더링할 센서를 결정합니다.

  4. IsaacReadSimulationTime 같은 OmniGraph 노드는 동일한 시뮬레이션 시간을 직접 읽거나, 기준 프레임 RationalTime이 주어진 경우 isaacsim.core.simulation_manager에서 유지하는 TimeSampleStorage 링 버퍼에서 보간하여 읽습니다.

When useFixedTimeStepping=true (the full Isaac Sim GUI default)#

physics_dt > loop_dt 경우에서, 물리 스텝이 실행되지 않는 프레임은 /ExternalSimulationTime을 변경하지 않습니다. 렌더 파이프라인은 여전히 매 앱 업데이트마다 실행되지만, 센서당 틱 카운터는 진행되지 않으며 해당 프레임에서 새 출력을 생성하는 센서가 없습니다. 물리가 마침내 스텝을 밟으면, 프림은 physics_dt만큼 앞으로 점프하고 예정된 센서들이 해당 프레임에서 렌더링됩니다.

sim_advance_per_wall_sec  =  (1 / timeCodesPerSecond)  *  loop_hz_wall

/app/player/useFixedTimeStepping이 false일 때(독립형 Python의 기본값) 위 표에 설명된 서브스텝-따라잡기 동작이 적용됩니다. 전체 Isaac Sim GUI 앱은 source/apps/isaacsim.exp.full.kit에서 이 carb 설정을 true로 설정합니다. true일 때, 타임라인은 실행 루프의 측정된 dt를 무시하고 Timeline::update() 내의 허용된 업데이트당 dt = 1 / timeCodesPerSecond를 강제합니다. 그러면 센서와 타임라인 시간은 다음과 같이 진행됩니다:

  • 결과:

  • loop_hz_wall == timeCodesPerSecond(기본값 둘 다 60Hz)이면, sim/wall = 1.0이고 시스템은 실시간으로 실행됩니다. 이것이 단독으로 호출될 때 isaacsim.core.rendering_manager.RenderingManager.set_dt()가 구성하는 것입니다 — rateLimitFrequency, targetFramerate, 및 timeCodesPerSecond를 동일한 값으로 정렬합니다.

  • loop_hz_wall < timeCodesPerSecond(예: 루프가 타임라인의 틱당 속도 아래로 속도 제한됨)이면, 시뮬레이션은 loop_hz_wall / timeCodesPerSecond 비율로 슬로우 모션으로 실행됩니다. /ExternalSimulationTime으로 게이팅된 RTX 센서는 비례적으로 느리게 발행합니다; OnPlaybackTick 기반 발행자(/clock, OmniGraph 틱)는 여전히 loop_hz_wall에서 발동됩니다. 이것이 사용자가 timeCodesPerSecond를 업데이트하지 않고 /app/runLoops/main/rateLimitFrequency만 설정할 때의 함정입니다: 물리 스텝 시간과 벽시계가 명확하지 않은 방식으로 분리됩니다.

    • loop_hz_wall > timeCodesPerSecond이면, 동작은 /app/player/useFastMode(timeline.set_play_every_frame(...)로도 노출됨)에 따라 달라집니다:

    • useFastMode = false (기본값): 타임라인은 서브샘플링하여(N = ceil(targetFramerate / timeCodesPerSecond)번째 실행 루프 틱마다 시간 진행) 런 루프가 더 빠르게 틱을 발생시키는 동안 타임라인이 구성된 속도를 유지합니다. 이는 시뮬레이션 시간을 빠르게 하지 않고 더 높은 렌더 제품 / OnPlaybackTick 속도를 구동하는 데 사용됩니다.

useFastMode = true(예: timeline.set_play_every_frame(True)가 호출될 때 설정됨 — 일부 Replicator 예제 및 벤치마크가 이를 수행함): 서브샘플 게이트가 우회됩니다. 모든 런루프 틱이 타임라인을 1 / timeCodesPerSecond만큼 진행시키므로, 시뮬레이션 시간은 loop_hz_wall / timeCodesPerSecond 비율로 벽시계 시간보다 빠르게 실행됩니다(빨리 감기). 이는 벽초당 가능한 많은 시뮬레이션 시간을 수집하려는 오프라인 데이터 생성을 위한 의도된 동작이지만, 시뮬레이션 시간이 벽시계 시간을 추적한다고 가정하는 코드를 깨뜨립니다.

Practical implications#

  • Setting the Simulation Rate (Advanced)의 참조 예제는 isaacsim.core.simulation_manager.SimulationManager.setup_simulation()과 isaacsim.core.rendering_manager.RenderingManager.set_dt()를 함께 사용하여 세 클럭 모두를 일관되게 유지합니다 — 위의 모든 모드를 구조적으로 방지합니다. useFastMode는 기본값인 false로 유지됩니다.

  • SimulationManager.get_simulation_time(), IsaacReadSimulationTime, 또는 프레임당 getSimulationTimeAtTime(rtime) 조회를 통해 항상 시뮬레이션 시간을 읽으십시오. 이것들은 렌더러도 보는 물리 기반 값을 반환하므로, 센서 스탬프, TF 스탬프 및 다운스트림 소비자가 일관성을 유지합니다.

  • 센서 또는 메시지 타임스탬프로 omni.timeline 현재 시간을 사용하지 마십시오. physics_dt > loop_dt 구성에서 두 클럭은 매 프레임마다 드리프트하며 물리 스텝에서만 재동기화되어 일관되지 않은 스탬프를 생성합니다.

Settings Reference#

물리, 렌더러, 및 다운스트림 파이프라인이 모두 프레임당 한 번씩 진행되도록 유지하려면, SimulationManager.setup_simulation(dt=1.0 / hz)와 RenderingManager.set_dt(1.0 / hz)를 통해 physics_dt == loop_dt로 설정하십시오. isaacsim.exp.base.zero_delay.kit 경험은 이를 구성하며 결정론적 단일 프레임 TF / 이미지 페어링을 위한 참조 설정입니다.

Note

Migration from Previous Releases#

6.0에서 Isaac Sim 5.x의 매 프레임 렌더 동작을 재현하려면(예: 회귀 디버깅), --/rtx/hydra/supportMultiTickRate=false로 실행하십시오. 6.0의 대부분의 다른 코드 경로는 전역 기본값을 가정하며 해당 설정이 비활성화된 상태로 검증되지 않았습니다.

General changes#

  1. 멀티틱 렌더링이 기본적으로 활성화되지 않은 릴리스에서 업그레이드하는 경우, 다음 변경 사항이 워크플로우에 영향을 미칠 수 있습니다.

  2. 멀티틱 렌더링과 함께 작동하도록 Camera 및 OmniSensor 프림을 업데이트하십시오. Camera 프림에 OmniSensorAPI 스키마를 적용하십시오. 이 스키마는 이미 OmniLidar/OmniRadar 프림에 기본적으로 적용되어 있습니다. 렌더 빈도를 제어하려면 omni:sensor:tickRate 속성을 설정하십시오. 멀티틱 렌더링은 기본 omni:sensor:tickRate가 0(자동트리거, 매 프레임 렌더링)인 경우 투명합니다.

  3. USD 애셋이 업데이트되었습니다. 기본 제공 센서 애셋에는 이제 OmniSensorAPI 스키마가 적용되어 있습니다. Camera 또는 OmniLidar/OmniRadar 프림이 있는 커스텀 USD 애셋이 있는 경우, OmniSensorAPI 스키마를 적용하고 omni:sensor:tickRate를 설정하여 렌더 빈도를 제어하십시오.

  4. frameSkipCount가 더 이상 사용되지 않습니다. ROS2/UCX/HSB 헬퍼 노드에서 frameSkipCount 사용을 센서 프림의 omni:sensor:tickRate로 교체하십시오. frameSkipCount Deprecation을 참조하십시오.

  5. RTX Lidar 누적이 USD 속성으로 이동되었습니다. Lidar 스캔 누적은 이제 OmniLidar 프림의 omni:sensor:Core:accumulateOutputs 속성으로 제어됩니다. 더 이상 사용되지 않는 isaacsim.sensors.rtx 확장의 IsaacExtractRTXSensorPointCloudNoAccumulator 어노테이터와 IsaacCreateRTXLidarScanBuffer 및 IsaacComputeRTXLidarFlatScan 노드들은 이 속성을 읽도록 업데이트되었습니다. 최신 IsaacExtractRTXSensorPointCloud 어노테이터와 OmniGraph 노드는 isaacsim.sensors.rtx.nodes에 있으며 accumulateOutputs에 따라 GMO 버퍼에 이미 전체 스캔 또는 프레임당 부분 스캔이 포함되어 있다고 가정합니다.

    단일 렌더 제품 대기를 전체 앱 업데이트로 교체하십시오. omni.syntheticdata.sensors.next_render_simulation_async 헬퍼(및 단일 렌더 제품을 대상으로 하는 다른 헬퍼)는 멀티틱 렌더링 하에서 센서당 틱 카운터를 올바르게 진행하지 않습니다. 대신, 전체 애플리케이션 업데이트 단계를 수행하고 모든 센서 틱이 처리되도록 보장하는 isaacsim.core.experimental.utils.app.update_app_async를 사용하십시오.

    import omni.syntheticdata

# Example values for an attached render product and a per-product wait count.
render_product_path = "/Render/RenderProduct_Replicator"
N = 1

# Deprecated: does not advance per-sensor tick counters under multi-tick rendering.
await omni.syntheticdata.sensors.next_render_simulation_async([render_product_path], N)

    이전:

    import isaacsim.core.experimental.utils.app as app_utils

# Number of full application update steps to perform.
N = 1

await app_utils.update_app_async(steps=N)

frameSkipCount Deprecation#

이후:

이전 릴리스에서는 ROS2 및 UCX 헬퍼 노드의 발행 속도가 각 헬퍼 노드의 frameSkipCount 입력으로 제어되었습니다. 이 파라미터는 이제 더 이상 사용되지 않습니다.

멀티틱 렌더링이 전역적으로 활성화된 경우, 센서가 데이터를 발행하는 빈도를 제어하는 올바른 방법은 센서 프림 자체에 omni:sensor:tickRate를 설정하는 것입니다. 이는 건너뛴 틱 동안 센서가 전혀 렌더링하지 않기 때문에 더 효율적입니다. 렌더링 후 출력을 버리는 것이 아닙니다.

frameSkipCount 파라미터는 이전 버전과의 호환성을 위해 여전히 작동하지만, 0이 아닌 값을 사용하면 사용 중단 경고가 기록됩니다. 향후 릴리스에서 제거될 예정입니다.

  • 사용 중단은 이전에 frameSkipCount를 노출했던 모든 헬퍼 노드에 적용됩니다:

  • ROS2 Camera Helper (isaacsim.ros2.bridge.ROS2CameraHelper)

  • ROS2 Camera Info Helper (isaacsim.ros2.bridge.ROS2CameraInfoHelper)

  • ROS2 RTX Lidar Helper (isaacsim.ros2.bridge.ROS2RtxLidarHelper)

UCX Camera Helper (isaacsim.ucx.bridge.UCXCameraHelper)

Multi-tick scheduling migration#

더 최신의 ROS2 RTX Radar Helper (isaacsim.ros2.bridge.ROS2RtxRadarHelper)는 이 사용 중단 이후에 도입되었으며 frameSkipCount를 전혀 노출하지 않습니다.

아래 표에는 5.x 속도 제어 입력과 각 센서 유형에 대한 권장 6.0 교체 방법이 나열되어 있습니다. 더 광범위한 확장 API 마이그레이션에 대해서는 RTX Sensors를 참조하십시오. 더 이상 사용되지 않는 isaacsim.sensors.camera 확장에 대해서는 Camera Sensors를 참조하십시오. auxOutputType -> GenericModelOutput 채널에 대해서는 Auxiliary Output Level and the GenericModelOutput RenderVar를 참조하십시오.

Known Issues#

RTX Radar autotriggers in 6.0 GA#

RTX Radar는 의도적으로 생략되었습니다: omni:sensor:tickRate는 6.0 GA의 OmniRadar에서 무시됩니다(RTX Radar autotriggers in 6.0 GA 참조). 대신 다운스트림에서 Radar 발행 속도를 게이팅하십시오.

Isaac Sim 6.0 GA에서, RTX Radar 렌더러는 OmniRadar 프림의 omni:sensor:tickRate를 무시하고 매 시뮬레이션 프레임마다 렌더링합니다. isaacsim.sensors.experimental.rtx.Radar에서 tick_rate를 설정해도 실제 렌더 간격에 영향이 없습니다; 속성은 여전히 프림에 작성되지만, 멀티틱 스케줄러는 Radar 프림을 건너뜁니다. 이는 향후 릴리스에서 수정될 예정입니다.

OmniLidar partial-scan fallback#

자동트리거 제한은 OmniLidar 또는 Camera 프림에는 영향을 미치지 않습니다.

Radar + Lidar frames-in-flight race#

OmniLidar 프림에서 omni:sensor:tickRate가 omni:sensor:Core:scanRateBaseHz와 같지 않으면, 센서는 매 프레임 부분 스캔을 방출하는 폴백으로 전환됩니다. 자세한 내용과 권장 해결책은 OmniLidar Tick Rate Must Equal scanRateBaseHz를 참조하십시오.

[Fatal] [carb.crashreporter-breakpad.plugin] Crashing: SIGFPE
at rtx.sensors.lidar.core.plugin::LidarRotary::openTrace

RTX Radar, RTX Lidar 및 Motion BVH를 결합한 씬에서 시뮬레이션 시작 후 처음 1~2 벽시계 초 동안 rtx.sensors.lidar.core.plugin에서 치명적 충돌이 발생할 수 있습니다. 충돌은 Lidar의 프레임당 추적이 센서 프로필이 초기화되기 전에 시작되는 RTX 센서 프레임워크의 frames-in-flight(FIF) 스케줄링의 타이밍 의존적 경쟁으로 인해 발생합니다. 영향을 받는 구성은 결정론적으로 충돌합니다; 영향을 받지 않는 하드웨어에서는 문제가 발생하지 않습니다. 오류는 LidarRotary::openTrace 내부의 부동 소수점 예외 또는, 드물게 v3.0 센서 스케줄러의 세그멘테이션 폴트로 나타납니다:

Standalone Python workaround#

시뮬레이션이 충돌 없이 약 1~2 벽시계 초 동안 실행되면, 세션은 나머지 수명 동안 안정적입니다.

from isaacsim import SimulationApp

simulation_app = SimulationApp({"headless": True, "enable_motion_bvh": True})

import carb
import numpy as np
import omni
from isaacsim.core.experimental.objects import Cube
from isaacsim.sensors.experimental.rtx import (
    Lidar,
    LidarSensor,
    Radar,
    RadarSensor,
    parse_generic_model_output_data,
)

Cube("/cube", sizes=2.0, positions=np.array([10.0, 0.0, 0.0]))

# Lidars: full wrap pre-play (USD prim + render product + annotators).
lidar_sensor_1 = LidarSensor(Lidar("/lidar_1"), annotators=["generic-model-output"])
lidar_sensor_2 = LidarSensor(Lidar("/lidar_2"), annotators=["generic-model-output"])

# Radar: USD authoring object only. Defer the RadarSensor wrap until after the
# Lidar frames-in-flight slots have stabilized post-play.
radar = Radar("/radar")

# Start playback and let the Lidars warm up for a few frames before wrapping
# the Radar. 5 frames is one full rotation of the default 3-slot
# frames-in-flight buffer plus a small margin; heavier scenes may need more.
timeline = omni.timeline.get_timeline_interface()
timeline.play()
for _ in range(5):
    simulation_app.update()

# Now safe to wrap the Radar. This call creates the Radar's render product and
# binds annotators - the operation that would open the FIF race window if done
# concurrently with Lidar attachment.
radar_sensor = RadarSensor(radar, annotators=["generic-model-output"])

# Continue running and verify every sensor is producing output.
for _ in range(60):
    simulation_app.update()

    for sensor, name in [
        (lidar_sensor_1, "lidar_1"),
        (lidar_sensor_2, "lidar_2"),
        (radar_sensor, "radar"),
    ]:
        data, _ = sensor.get_data("generic-model-output")
        gmo = parse_generic_model_output_data(data)
        carb.log_warn(f"{name}: numElements={gmo.numElements}")

timeline.stop()
simulation_app.close()

독립형 Python 워크플로우에서는 frames-in-flight가 안정화된 후까지 Radar의 렌더 제품 생성과 어노테이터 또는 라이터 연결을 지연하십시오. timeline.play() 이전에 Lidar를 정상적으로 구성하되, Radar의 USD 작성 객체만 재생 전에 구성하고 짧은 워밍업 윈도우 이후로 RadarSensor 래핑을 연기하십시오:

OmniGraph workaround#

5프레임 워밍업은 보수적입니다: 기본 3슬롯 frames-in-flight 버퍼의 전체 회전 더하기 소량의 여유입니다. 더 무거운 씬에서는 더 큰 값이 필요할 수 있습니다.
